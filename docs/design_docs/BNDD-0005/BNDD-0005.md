## Purpose

This document turns the ClickHouse direction from [BNDD-0002](../BNDD-0002/BNDD-0002.md) into an implementation-facing repository decision for crawler persistence.

The goal is not to design a generic data platform.

The goal is to choose a maintainable first architecture that:

- persists crawler observations durably in ClickHouse
- supports one active crawl run with high internal concurrency
- exposes a narrow control/query boundary for desktop, web, and future agents
- keeps the crawler append-first and analytics-oriented
- treats security as part of the design, not as a later patch

## Decision Summary

The repository will adopt this direction for crawler persistence and control:

- use `ClickHouse` as the default database for crawler observations and run tracking
- introduce a dedicated ClickHouse repository crate in the workspace
- keep protocol/session code in `crates/btc-network` free of ClickHouse-specific logic
- define a shared mockable repository contract in the shared Rust layer
- keep v1 explicitly `single-instance` for crawler execution and control ownership
- keep `one active run in the coordinator process`, while preserving high worker concurrency inside that run
- batch-write crawler observations continuously during the run rather than waiting for a later bulk-ingest phase
- track run state through append-only run checkpoints
- use stable observation and batch identifiers so retries and duplicate detection stay explicit
- model protocol-derived observation data separately from IP-enrichment data before persistence
- keep observation assembly and run-state decisions in the shared Rust layer, while keeping network I/O, MMDB lookups, batching, and ClickHouse access in the imperative shell
- derive the first current-state analytics query with `argMax` at query time instead of adding a separate latest-state table in v1
- use checked-in forward-only SQL migrations as the default schema-management workflow
- use the official ClickHouse Rust client over HTTP with long-lived process-local client reuse
- expose only narrow start/stop/status/query operations to product surfaces and future agent/tool callers

## Why This Decision Exists

The crawler already exists, but it is still in-memory only.

That is the main gap between the current implementation and the observability direction described in [BNDD-0001](../BNDD-0001/BNDD-0001.md).

The crawler workload has a few properties that shape the decision:

- observations are append-heavy
- historical analytics matter more than transactional updates
- the product needs progress tracking for long-running crawls
- the first useful read path is analytical, not relational CRUD
- security matters at the control plane because crawl execution is a networked action

[BNDD-0002](../BNDD-0002/BNDD-0002.md) already settled the database choice for this workload.

This document settles the next layer:

- how ClickHouse should be integrated
- how schemas should be managed
- how crawler runs should be controlled
- how the same contract should reach desktop, web, and future API consumers safely

## Architecture Boundaries

The crawler persistence stack should keep the repository boundaries explicit:

```text
React / Desktop UI / Future agents
  -> app-facing client contract
  -> Tauri commands or HTTP API
  -> shared crawler workflow in crates/btc-network
  -> enrichment provider trait
  -> repository trait
  -> MMDB-based enrichment adapter
  -> ClickHouse adapter crate
  -> ClickHouse
```

Rules:

- do not put ClickHouse queries in React components, Tauri commands, or CLI entrypoints
- do not move protocol parsing or session logic into the repository crate
- do not move ASN, country, or prefix derivation into React components, Tauri commands, or ClickHouse queries
- keep the ClickHouse implementation behind a trait so the shared crawler workflow can be tested with mocks
- keep IP enrichment behind a trait so the shared workflow can be tested without filesystem-backed MMDB state
- treat the future HTTP API and the desktop shell as adapters over the same workflow boundary, not as separate crawler implementations

### Crate direction

The intended split is:

- `crates/btc-network`: shared crawler types, workflow orchestration, validation, pure observation assembly, enrichment trait, and repository trait
- `crates/btc-network-clickhouse`: ClickHouse schema, migrations, repository implementation, and ClickHouse-specific query code

This keeps the protocol-first crate focused and avoids dragging ClickHouse dependencies through unrelated modules.

### Functional core and imperative shell

This design fits a functional-core / imperative-shell split well.

The shared Rust layer should own pure decisions such as:

- how raw crawler results become normalized observations
- which endpoints are eligible for IP enrichment
- how enrichment results are represented
- how a persisted observation is assembled from raw data plus enrichment
- how crawl phases and validation rules behave

The imperative shell should own effectful concerns such as:

- peer connections and Bitcoin session I/O
- clocks and identifier generation
- MMDB loading and IP metadata lookups
- channels, batching, retries, and task orchestration
- ClickHouse writes and analytics reads

## Repository Contract

The shared Rust layer should define a repository boundary with explicit domain types.

The first public concepts are:

- `CrawlRunId`
- `CrawlPhase`
- `CrawlRunCheckpoint`
- `RawNodeObservation`
- `IpEnrichment`
- `PersistedNodeObservation`
- `StartCrawlRequest`
- `StopCrawlRequest`
- `CountNodesByAsnRow`
- `IpEnrichmentProvider`
- `CrawlerRepository`

The repository trait starts with these operations:

- `insert_observation`
- `insert_observations_stream`
- `insert_run_checkpoint`
- `get_run_checkpoint`
- `list_runs`
- `count_nodes_by_asn`

The repository-facing observation model should also carry stable identifiers:

- `observation_id`
- `batch_id`

Identifier rules:

- `observation_id` is generated once for the logical observation before it enters the writer pipeline
- retries for the same logical observation reuse the same `observation_id`
- `batch_id` is generated once for the writer batch before the first flush attempt
- retries for the same writer batch reuse the same `batch_id`
- both identifiers only need to be unique within the run

### Why both single and streamed insert paths exist

`insert_observations_stream` is the crawler hot path.

It exists because ClickHouse performs best when rows are inserted in larger client-side batches rather than as many tiny inserts.

`insert_observation` still exists because it is useful for:

- tests
- small admin flows
- low-volume control-plane writes
- isolated integration probes

It is not the intended write path for high-volume crawler observations.

## Observation Assembly and Enrichment

The happy path needs an explicit boundary between what the Bitcoin protocol can tell us and what the crawler must derive locally.

### Field provenance

The Bitcoin P2P protocol and local crawler runtime can directly provide fields such as:

- endpoint
- network type
- handshake status
- protocol version
- services
- user agent
- start height
- relay flag
- discovered count
- latency
- failure classification

Those fields are protocol-derived or locally measured.

The protocol cannot provide:

- ASN
- ASN organization
- country
- prefix

Those fields are enrichment-derived and must come from a separate IP metadata source.

### Happy path

The intended v1 pipeline is:

1. crawl a peer and collect the raw protocol-derived result
2. normalize the endpoint and network type into a `RawNodeObservation`
3. if the endpoint is routable IPv4 or IPv6, resolve IP metadata through `IpEnrichmentProvider`
4. assemble a `PersistedNodeObservation` from raw observation plus optional enrichment
5. enqueue the persisted observation into the writer pipeline
6. batch-write rows and checkpoints continuously while the run is active

This keeps enrichment in the same streaming pipeline as persistence without introducing a separate delayed backfill stage in v1.

### Representation rules

The domain model should keep raw observation and enrichment distinct even when the final storage row merges them.

Expected v1 behavior:

- IPv4 and IPv6 endpoints may have enrichment attached when lookup succeeds
- Tor and other overlay-network endpoints are not geolocated like normal IP addresses
- non-routable or unsupported endpoints should keep enrichment fields empty rather than synthesizing fake geographic data
- lookup failure should be representable separately from `not applicable` so the pipeline stays observable

This means the persisted row should preserve both:

- the protocol truth
- the enrichment outcome

without pretending that missing enrichment is the same thing as protocol data.

## Storage Model

The first physical model should stay small and explicit.

### node_observations

This is the append-only fact table.

It should preserve the BNDD-0001 observation shape, including the fields needed for early analytics and enrichment, such as:

- observation timestamp
- crawl run id
- observation id
- batch id
- endpoint
- network type
- handshake status
- confidence level
- protocol version
- services
- user agent
- start height
- relay flag
- discovered count
- latency
- ASN
- country
- prefix
- optional failure classification for failed attempts

The semantics of those fields should stay explicit:

- `ASN`, `country`, and `prefix` are enrichment-derived, not protocol-derived
- they should be nullable/optional in the shared model and stored in a way that preserves `not applicable` or `unavailable`
- overlay-network endpoints should rely on `network_type` rather than fake geographic values

This table should use monthly partitioning by observation time as the default production-oriented layout, consistent with BNDD-0002.

### `crawler_run_checkpoints`

This is an append-only checkpoint table for crawler runs.

Each row is a durable recovery point, not an update-in-place record.

That means a run writes:

- an initial checkpoint
- periodic checkpoints while running
- a final checkpoint on completion or failure

Useful checkpoint fields include:

- run id
- phase
- checkpoint timestamp
- started at
- stop reason or outcome
- scheduled tasks
- successful handshakes
- failed tasks
- queued nodes total
- unique nodes
- discovered node states
- persisted observation rows
- writer backlog or drain indicators
- serialized frontier state needed to resume the run
- caller/source metadata for auditability

This design fits ClickHouse better than hot-path row updates and also leaves a durable progress history for UI/API reads.

The important semantic distinction is:

- a `snapshot` is only a view of progress
- a `checkpoint` is a view of progress plus enough durable state to resume from it

This document chooses `checkpoint` because resumability is part of the design.

### `schema_migrations`

This is a small ledger table used by the migration runner.

It should record:

- version
- name
- checksum
- applied timestamp

### First analytics query

The first analytics read is:

- `how many latest verified nodes by ASN`

This should be implemented from `node_observations` with query-time `argMax` semantics per endpoint.

The ordering key should be stable, not timestamp-only.

The intended pattern is to use a tuple such as:

- `(observed_at, observation_id)`

That keeps the initial design simple:

- raw observations are stored once
- current state is derived when queried
- a dedicated latest-state materialization is deferred until read pressure justifies the extra complexity

This is consistent with BNDD-0001 logically and BNDD-0002 operationally.

## Crawler Run State Machine

The control-plane rule is:

- v1 is single-instance only
- one active crawl run in the coordinator process

That rule is about control ownership, not about reducing crawl throughput.

Inside the active run, the crawler should remain highly concurrent.

This should be treated as an explicit state-machine decision.

The active run is owned by one in-process coordinator.

That coordinator is the only authority allowed to:

- accept `startCrawl`
- accept `stopCrawl`
- create a new active run
- transition run phases

This keeps start/stop races serialized inside one process and avoids pretending that v1 already solves distributed coordination.

### Phases

Each run moves through these phases:

- `bootstrap`
- `crawling`
- `draining`
- `completed`
- `failed`

```text
bootstrap -> crawling
bootstrap -> draining
bootstrap -> failed

crawling -> draining
crawling -> failed

draining -> completed
draining -> failed
```

A stop request transitions the run to `draining`.

If the stop arrives during `bootstrap`, the coordinator records the stop request and moves to `draining` as soon as bootstrap completes enough to shut down cleanly.

### Why not separate crawling and persistence into fully distinct phases

The crawler may discover a large frontier quickly.

If the system waits until the crawl finishes before it starts persisting observations, it creates unnecessary memory pressure and a second backlog problem.

The correct first design is a pipeline:

1. resolve seed nodes and create the run id
2. start a bounded Tokio worker pool for node fetches
3. build raw protocol-derived observations from crawl results
4. normalize and enrich eligible observations before enqueueing them into a bounded channel
5. let a dedicated writer task batch observations and flush them continuously
6. persist checkpoints while the run is active
7. on shutdown, stop new work, drain pending batches, and then write the final checkpoint

This keeps throughput high without turning persistence into a delayed second stage.

### Queue model

This is not a work-stealing design.

The initial model is:

- one coordinator process
- one shared frontier queue
- bounded worker concurrency
- one writer pipeline
- one active run state machine

Workers keep pulling from the shared frontier until the crawler reaches a stop condition.

The design accepts repeated observations across different runs in exchange for simpler coordination.

It does not accept unbounded repeated attempts inside the same run.

### Concurrency versus parallelism

This document uses `concurrency` intentionally.

The crawler will be coordinated by the Tokio multi-thread runtime:

- many node visits can be in flight concurrently
- work may execute in parallel when runtime threads are available
- the important control is bounded in-flight work, not maximum thread creation

The concurrency setting should be an explicit runtime control.

### Single-instance boundary

V1 should state this boundary explicitly:

- the crawler runtime is single-instance only
- the coordinator and crawler worker runtime live in the same process
- the design should keep adapter and workflow boundaries clean enough that the API surface can later be split across multiple instances if needed
- that future split does not imply that crawler ownership becomes multi-instance in v1

This document intentionally does not choose a future distributed coordination mechanism.

### Checkpoint contents

Because resumability is required, the checkpoint must carry enough state to rebuild the in-memory crawler state.

For v1 that means the checkpoint should include, in serialized form:

- the pending frontier
- the deduplication state needed to avoid re-queuing already known endpoints inside the run
- the run counters already listed above

This is intentionally simple and single-process oriented.

The document does not choose a more complex distributed queue or lease-based work model for v1.

### Progress tracking

Progress is tracked from the run state machine plus periodic checkpoints.

Each checkpoint should include at least:

- current phase
- frontier size
- in-flight work count
- scheduled tasks
- successful handshakes
- failed tasks
- persisted observation rows
- writer backlog

Checkpoints should be emitted:

- when the run starts
- on each phase transition
- periodically while `crawling`
- once more after `draining` completes

## Failure, Retry, and Duplication Semantics

Failure and duplication behavior must be explicit.

### Node visit retries

The default v1 rule is:

- no automatic same-run retry for a node visit that already failed

Reason:

- it avoids hot-looping bad endpoints
- it reduces duplicate observations inside the run
- the next crawl run already provides a natural retry window

### Batch write retries

Observation batch writes should retry with:

- bounded retries
- exponential backoff
- the same `batch_id`
- the same `observation_id` values

If those retries are exhausted, the run should transition to `failed`.

### Process failure in the middle of a run

If the process exits in the middle of a run:

- the last persisted checkpoint remains the durable recovery point
- work performed after that checkpoint may be lost
- work performed after that checkpoint may also be replayed if it reached ClickHouse but was not yet reflected in the checkpoint

At coordinator startup, before creating a new run, the process should try to recover the latest active run from its last checkpoint.

Recovery rules:

- if the latest checkpoint is `bootstrap`, `crawling`, or `draining`, rebuild the in-memory crawler state from that checkpoint and continue the same run
- if the checkpoint payload is missing or cannot be restored safely, write a terminal checkpoint for that run with `phase = failed`
- only after recovery succeeds or the failed terminal checkpoint is written may a new run be created

That startup recovery is what restores the one-active-run guarantee after a crash.

This means repeated observations after recovery are acceptable.

They are a consequence of resuming from the last durable checkpoint boundary, not corruption.

### Duplicate handling

The design should distinguish two cases:

- the same logical observation inserted more than once because of retry ambiguity
- the same endpoint observed again in a later crawl run

The second case is expected and valid.

The first case is undesirable but must remain identifiable.

That is why each row carries:

- `run_id`
- `observation_id`
- `batch_id`

This keeps later cleanup or query-side deduplication possible if it becomes necessary.

The document does not require physical deduplication in ClickHouse for v1.

V1 only requires that duplicates caused by retry ambiguity or checkpoint-boundary replay remain identifiable.

### Latest-row identification

Queries that need the latest logical row should not rely on `observed_at` alone.

They should use a stable ordering key such as:

- `(observed_at, observation_id)`

### One active run versus many active runs

This document rejects multiple concurrent runs as the initial control-plane model.

Reason:

- it increases write pressure and network load immediately
- it complicates UI state and operator expectations
- it makes run ownership, stop behavior, and analytics semantics harder to reason about
- the current product does not yet need simultaneous competing crawler jobs

If the repository later needs many active runs, it should revisit that explicitly rather than adding it by default now.

## Schema Management Strategy

The default schema-management workflow is checked-in forward-only SQL migrations applied explicitly by an admin or deployment action and recorded in a ClickHouse migration ledger table.

Plain SQL wins now because it keeps the schema close to real ClickHouse DDL and avoids adding a second abstraction layer too early.

Atlas remains a later revisit option if migration linting, drift detection, or multi-environment governance become materially painful without tooling.

### Operational rules

- do not auto-run migrations from React, desktop startup, crawler start, or future agent prompt paths
- run migrations as an explicit admin/deploy step
- prefer forward-only changes
- use timestamped migration filenames in `YYYYMMDDHHMMSS_slug.sql` form so creation order is obvious and lexicographic order stays stable
- document backfills or manually coordinated steps next to the migration when a schema change is not purely online-safe
- treat DDL privileges as separate from normal application write privileges

## Connection and Session Strategy

The ClickHouse access defaults are:

- use the official Rust client over HTTP
- keep one long-lived client per process and clone/reuse it rather than opening throwaway connections for each operation
- do not put an external pooler in front of ClickHouse in v1
- do not rely on `session_id` for repository correctness or authorization semantics

### Why this differs from a typical Postgres setup

Postgres applications often benefit from a pooler such as pgBouncer because many short-lived transactional connections are normal and session management is part of ordinary application behavior.

The first ClickHouse design in this repository is different:

- writes are append-heavy and batch-oriented
- reads are narrow approved analytics queries
- correctness should not depend on sticky server-side session state
- process-local client reuse plus bounded writer concurrency is enough for the initial scale

`chproxy` remains a later option if the repository eventually needs:

- HTTP load balancing
- stronger query shaping or rate limits in front of ClickHouse
- multi-node routing

It is not a day-one requirement.

## Security Defaults

Keep the security section short and explicit.

### Role separation

ClickHouse privileges should be split by responsibility:

- migration role: DDL only
- application writer role: insert observations and run checkpoints
- analytics reader role: approved read paths only

### Explicit rejections

This design rejects exposing:

- raw SQL
- arbitrary ClickHouse settings
- shell-like execution paths
- direct database credentials to UI clients

### Input validation

The shared workflow and its adapters must validate:

- `startCrawl` configuration values
- `stopCrawl` and `getCrawlRun` identifiers
- pagination and filter inputs for analytics reads

### Prompt-injection and supply-chain rule

Future agent/tool integrations must treat prompts, web content, repository content, and remote peer data as untrusted input.

They must not expand into prompt-to-shell, prompt-to-SQL, or prompt-to-tool execution paths.

Any dependency, proxy, migration tool, or automation added in this path must also follow the repository’s suspicious-package rule.

## Web, Desktop, and Future API Contract

The product contract should be shaped once and reused across adapters.

### Shared contract

The app-facing client contract should expose:

- `startCrawl`
- `getCrawlRun`
- `listCrawlRuns`
- `countNodesByAsn`
- `stopCrawl`

### Desktop

Desktop should stay a thin Tauri shell over shared Rust workflow code.

It should not implement its own crawler logic and should not embed ClickHouse details in Tauri commands.

### Web

The web runtime needs a minimal HTTP API for real crawler control and analytics reads.

That requirement refines the timing of the future API direction from BNDD-0003, but it does not change the deployment shape chosen there.

### Future API

The future Rust API should expose the same narrow contract and keep:

- read endpoints separate from control endpoints
- stronger authorization on control endpoints than on analytics reads
- rate limiting and request shaping at the API edge

## Test Plan

The first implementation guided by this BNDD should include:

Integration tests should be the default in this flow because the important behavior crosses runtime, persistence, and control-plane boundaries.

Use unit tests only for small pure helpers such as state-transition validation or identifier generation.

### Integration tests

- migrations apply correctly and are recorded in the ledger
- streamed batch inserts succeed and bounded retries do not lose row identity
- a failed process can resume the same run from the last durable checkpoint
- latest-verified-nodes-by-ASN returns the expected result under repeated observations
- one-active-run enforcement rejects duplicate starts
- checkpoints reflect phase transitions and draining behavior
- routable IPv4 and IPv6 observations enrich into ASN/country/prefix before persistence
- overlay-network or non-routable endpoints persist without fake geographic enrichment
- raw observation assembly remains correct when enrichment is unavailable or lookup fails

### Security tests

- invalid control/query input is rejected
- adapters do not expose raw SQL or shell-style execution
- role separation is enforced between migration, writer, and read-only access
- enrichment adapters do not turn repository, web, or peer-controlled text into tool or shell execution

## Tradeoffs and Revisit Conditions

This design is intentionally pragmatic.

### Accepted tradeoffs

- current-state analytics are computed at query time in v1
- one active run is simpler than a full concurrent job scheduler
- plain SQL migrations are simpler than adopting a schema-as-code tool immediately
- process-local ClickHouse client reuse is simpler than adding a proxy layer immediately

### Revisit conditions

Revisit this BNDD if one or more of these become true:

- latest-state queries become frequent enough to justify a dedicated materialized representation
- many concurrent crawler jobs become a real product need
- the schema-management workflow becomes painful without drift/linting tooling
- ClickHouse access needs stronger centralized proxy controls
- the browser-backed crawler API becomes substantial enough to require a separate BNDD for API shape and deployment detail
