## Purpose

Add first-version Sybil-oriented analytics derived from crawler-observable data.

The goal is not to claim direct Sybil detection from crawler observations alone. The goal is to expose explainable network-level signals that can help identify suspicious concentration, identity inflation, and coordinated-looking behavior patterns for further investigation.

This work is inspired by:

- J. R. Douceur, "The Sybil Attack," 2002
- P. Swathi, C. Modi, and D. Patel, "Preventing Sybil Attack in Blockchain using Distributed Behavior Monitoring of Miners," 2019

## Decision Summary

Adopt a conservative Sybil-oriented analytics slice:

- compute raw concentration and uniformity metrics from the latest crawler run
- expose heuristic signals separately from raw metrics
- never expose crawler-derived observations as confirmed attack claims
- keep the metrics explainable and sourceable to observed fields
- keep queries run-scoped and bounded by the selected latest run phase
- prefer direct aggregate SQL over row export or browser-side inference
- document NAT, VPN, Tor, cloud-hosting, geolocation, and incomplete-visibility limits in the API/docs/UI

The first version should add one read-only last-run report endpoint rather than spreading Sybil semantics across every existing distribution endpoint.

## Research Context

Douceur frames the Sybil problem as identity inflation: one real entity can present many identities when identity creation is cheap. A Bitcoin crawler cannot prove which network endpoints share one controlling entity, but it can show concentration and sameness patterns that are relevant to identity inflation.

Swathi, Modi, and Patel emphasize behavior monitoring and coordinated pattern analysis. The crawler does not observe miner behavior, consensus influence, or transaction propagation in this slice. It does observe handshake-level peer metadata, endpoint location/enrichment, and repeated snapshots. The implementation must stay inside that evidence boundary.

## Definitions

| Term | Meaning |
| --- | --- |
| Raw metric | Direct aggregate over crawler-observed fields, such as node count by ASN or top prefix share. |
| Heuristic signal | Explainable rule over raw metrics that says a pattern merits review. |
| Confirmed attack claim | A statement that a Sybil attack is occurring or that endpoints share an attacker. This project must not emit these claims from crawler data alone. |
| Cluster | Group of verified peers sharing an observed property, such as ASN, IP prefix, country, or software fingerprint. |
| Software fingerprint | Initial tuple of `protocol_version`, `services`, `user_agent`, and `relay` when available. |

## Metric Scope

Use only data the crawler already persists:

- `crawl_run_id`
- endpoint host, port, and network type
- protocol version
- services bitfield
- user agent
- start height
- relay flag
- discovered peer address count
- latency when available
- failure classification when available
- ASN
- ASN organization
- country
- IP prefix
- observation timestamp

Metrics should use verified observations for identity and software concentration unless a metric explicitly describes failures or reachability instability.

## Initial Metrics

| Metric | Type | Definition | Interpretation |
| --- | --- | --- | --- |
| `top_asn_share` | raw metric | Largest verified-node share held by one ASN in the selected latest run. | High values can indicate hosting concentration or possible identity concentration. Cloud and ISP concentration are common benign explanations. |
| `asn_hhi` | raw metric | Herfindahl-Hirschman index over verified-node counts by ASN. | Higher values mean more ASN concentration. It is not an attack probability. |
| `top_prefix_share` | raw metric | Largest verified-node share held by one observed IP prefix. | High values can indicate dense endpoint clustering. NAT, data centers, VPN exits, and coarse geolocation prefixes can inflate this. |
| `prefix_density` | raw metric | Verified peer count per observed IP prefix, with top prefixes returned. | Dense prefixes are review candidates when paired with software uniformity or repeated behavior. |
| `top_country_share` | raw metric | Largest verified-node share held by one country. | Useful context only; country concentration is too coarse for direct suspicion. |
| `top_software_fingerprint_share` | raw metric | Largest verified-node share for a shared software fingerprint. | High share can reflect popular clients, default settings, or coordinated deployment. |
| `cluster_fingerprint_uniformity` | heuristic signal | For top ASN and prefix clusters, share of peers that use the most common software fingerprint. | A concentrated cluster with unusually uniform software can merit review. |
| `cluster_height_uniformity` | heuristic signal | For top ASN and prefix clusters, share of peers with the same start-height bucket. | Similar heights can indicate coordinated startup or common deployment state, but also common release timing. |
| `repeated_observation_pattern` | heuristic signal | Repeated shared tuple across observations when data exists: software fingerprint, start-height bucket, relay flag, and discovered-peer-count bucket. | Repeated patterns can indicate coordinated configuration. It is weaker without multi-run trend evidence. |

Initial thresholds should be explicit constants with tests. They should be conservative and documented as review thresholds, not detection thresholds. Suggested first-version signal levels:

| Level | Meaning |
| --- | --- |
| `info` | Raw metric is noteworthy but common benign explanations dominate. |
| `watch` | Metric crosses a conservative threshold and should be tracked across runs. |
| `review` | Multiple related metrics cross thresholds in the same cluster, such as high prefix density plus high fingerprint uniformity. |

The API must not use `attack`, `malicious`, `compromised`, or equivalent labels for these signals.

## API Contract Direction

Add one endpoint:

```text
GET /api/v1/network/last-run/sybil-metrics?phase=<finished|any|comma-separated-phases>&limit=<n>
```

Response shape:

```json
{
  "runId": "uuid",
  "phase": "finished",
  "observedAt": "RFC3339",
  "verifiedNodeCount": 1234,
  "rawMetrics": {
    "topAsnShare": {
      "asn": 64512,
      "asnOrganization": "Example ASN",
      "nodeCount": 120,
      "share": 0.097
    },
    "asnHhi": 0.021,
    "topPrefixShare": {
      "prefix": "203.0.113.0/24",
      "nodeCount": 40,
      "share": 0.032
    }
  },
  "signals": [
    {
      "level": "watch",
      "kind": "cluster_fingerprint_uniformity",
      "clusterType": "asn",
      "clusterKey": "64512",
      "nodeCount": 30,
      "share": 0.85,
      "interpretation": "Many peers in this ASN share the same software fingerprint.",
      "confidenceBoundary": "This is a suspicious concentration signal, not proof of shared control or a Sybil attack."
    }
  ],
  "attackClaims": [],
  "limitations": [
    "Crawler data cannot prove operator identity.",
    "NAT, VPN, Tor, hosting providers, and incomplete crawler visibility can create benign concentration."
  ]
}
```

Rules:

- `attackClaims` must be present and empty in the first version.
- `limitations` must be present in every response.
- `signals[].confidenceBoundary` must be present for every heuristic signal.
- `phase` follows existing last-run distribution phase semantics.
- `limit` caps returned clusters/signals, not rows scanned.
- OpenAPI must document every metric definition and the non-detection boundary.

## Storage And Query Direction

Use PostgreSQL aggregation behind the crawler analytics trait:

```text
apps/api
  -> CrawlerAnalyticsReader::get_last_run_sybil_metrics
  -> btc-network-postgres aggregate queries
  -> node_observations + crawler_run_checkpoints
```

Dominant cost driver is `node_observations` rows for the selected run. Keep the first implementation run-scoped:

- select one latest run using the same phase filter semantics as last-run distributions
- aggregate by ASN, IP prefix, country, and software fingerprint in SQL
- compute HHI from grouped counts, not from raw rows in the API
- only pull bounded top clusters into Rust for signal construction
- do not introduce materialized views until measured query cost requires them

Index review should focus on existing access paths for:

- latest run selection by phase and checkpoint time
- `node_observations.crawl_run_id`
- grouping fields used by the first metrics

If grouping on software fingerprint becomes costly, prefer a generated or persisted fingerprint key only after measurement shows the query is a real bottleneck.

## Frontend Direction

Surface these metrics as evidence, not verdicts.

Preferred UI placement:

- keep `Risk` as a primary sidebar destination, not a hidden subview inside `Network Analytics`
- keep `Network Analytics` focused on snapshot analytics, with `Status` and `Crawler Runs` as page-level header toggles inside that area
- render Sybil-oriented signals as the first topic card in the `Risk` library
- treat the `Risk` library as a growable evidence surface: each topic owns its metric values, interpretation copy, and limitation boundaries
- show raw metric values beside any heuristic signal
- label sections with terms like `Concentration`, `Uniformity`, `Review signals`, and `Boundaries`
- avoid a single opaque Sybil score in the first version
- keep limitation copy inside each topic card or its future detail drawer

Long-term `Risk` library rules:

- each risk topic should be data-driven from a small descriptor model so filters, search, and pagination can be added without changing the page structure
- topic cards may summarize live inputs, planned BNDD-backed metrics, or documented limitations, but must clearly distinguish those states
- cards must remain evidence-oriented and educational; they should teach what crawler data can and cannot prove without becoming blog-style marketing content
- add pagination only when topic count or query cost makes it necessary
- add filtering only over explicit fields such as category, live/planned status, signal level, or metric family

The UI should not claim:

- "Sybil attack detected"
- "attacker"
- "malicious nodes"
- "controlled by one entity"

Acceptable wording:

- "Suspicious concentration signal"
- "Uniform cluster merits review"
- "Crawler-visible evidence only"
- "Not proof of shared control"

## Interpretation Rules

Implementation and docs must preserve three layers:

| Layer | Allowed Output | Disallowed Output |
| --- | --- | --- |
| Raw metrics | Counts, shares, HHI, top clusters, top fingerprints. | Suspicion labels without a heuristic rule. |
| Heuristic signals | `info`, `watch`, or `review` with rule explanation and confidence boundary. | Attack labels or claims of shared control. |
| Confirmed attack claims | Empty list in this version. | Any crawler-only assertion that a Sybil attack exists. |

Every heuristic signal should include:

- source metric names
- cluster type and key
- observed count
- observed share or uniformity ratio
- threshold that was crossed
- interpretation
- confidence boundary

## Limitations

Document these limitations in the BNDD, API docs, agent guide, and frontend copy:

- a crawler observes reachable network endpoints, not real-world identities
- one operator can use many ASNs or prefixes, and many operators can share one ASN or prefix
- NAT, carrier-grade NAT, VPNs, proxies, cloud hosting, and shared infrastructure can create benign concentration
- Tor onion peers intentionally hide IP-based location and prefix data
- ASN, country, and prefix enrichment can be stale, missing, or coarse
- popular node software can create benign fingerprint uniformity
- start-height similarity can reflect release timing, reboot timing, or common infrastructure
- crawler coverage is incomplete and time-dependent
- failed observations can reflect network conditions, not peer intent
- these metrics do not observe mining behavior, mempool behavior, transaction propagation, consensus influence, or social identity

## Out of Scope

This BNDD does not include:

- confirmed Sybil detection
- peer banning, enforcement, or automated mitigation
- miner-behavior monitoring
- transaction or mempool propagation analysis
- graph-topology inference beyond crawler observations
- opaque ML scoring
- active probing beyond the existing crawler behavior
- storing sensitive operator labels or manual accusations
- full historical trend analysis across many runs

Historical trends can be added later after the first run-scoped metrics are stable and tested.
