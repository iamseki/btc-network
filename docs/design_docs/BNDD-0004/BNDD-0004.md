## Purpose

This document proposes the local-development observability path for agentic work in this repository.

The intended flow is:

```text
Codex built-in OTEL export
  -> OpenTelemetry Collector
  -> Langfuse
```

The goal is not to design a full production observability platform.

The goal is to choose the simplest local setup that:

- makes Codex activity inspectable
- keeps the integration vendor-neutral
- uses OpenTelemetry as the transport boundary
- keeps a clear path to richer workflow tracing later
- avoids coupling Codex directly to a single backend

This direction is also intended to improve the cost effectiveness of agentic development.

Agentic workflows can waste time and tokens in ways that are hard to see without telemetry:

- repeated failed tool calls
- avoidable retries
- slow approval or tool execution paths
- prompt patterns that generate too much low-value work
- backend or transport failures that look like agent weakness instead of infrastructure problems

The purpose of this stack is therefore not only visibility.

It is also to create a feedback loop that helps reduce wasted developer time, wasted model usage, and unproductive debugging cycles.

## Decision framing

The repository is already using Codex as part of the development workflow.

That creates a practical observability need:

- see what Codex is doing during agentic sessions
- inspect failures in API calls, streaming, and tool execution
- keep local setup easy enough that it is actually used
- preserve a path to future team-wide or hosted observability without redesigning the stack

Two current facts shape the decision:

- Codex documents OpenTelemetry export via `[otel].exporter`
- Langfuse documents OTLP-over-HTTP ingestion and positions itself as an OpenTelemetry backend for LLM observability

There is one important nuance:

- the current Codex observability documentation is written in terms of structured OTEL event export and explicitly describes log-style event emission
- Langfuse documentation is trace-oriented and describes OTLP ingestion on `/api/public/otel` and `/api/public/otel/v1/traces`

That nuance is exactly why a collector is the right boundary.

The collector gives one stable integration point between Codex and Langfuse, even if signal mapping, filtering, fan-out, or future instrumentation changes are needed.

That boundary is also economically useful.

A collector-based design makes it easier to inspect where cost is actually being created:

- in Codex request patterns
- in tool usage frequency
- in retries or backend instability
- in slow or noisy parts of the workflow

Without that visibility, the team is more likely to respond to cost by guessing, reducing usage blindly, or over-optimizing prompts without understanding where waste originates.

## Constraints

The observability direction must respect these constraints:

- scope is local development first
- setup should be runnable on a single developer machine
- configuration should prefer Docker Compose and standard OTLP endpoints
- the default path should stay open-source and vendor-neutral
- initial privacy risk should stay low
- the first iteration should avoid speculative extra infrastructure
- the first iteration should not require repository-specific application tracing work

Two practical implications follow:

- Codex should be configured with the smallest useful OTEL surface first
- the collector should handle batching, memory limiting, and backend export from day one

## Tools in the stack

### Codex

Codex can emit OpenTelemetry-formatted observability data when `[otel].exporter` is enabled in `config.toml`.

Per the current Codex documentation, the exported data includes structured events around:

- conversation starts
- API requests
- SSE and WebSocket events
- tool decisions
- tool results
- optional user prompt logging

Codex is the producer in this design. It should not be the place where backend-specific routing logic lives.

### OpenTelemetry Collector

The OpenTelemetry Collector is the transport and processing layer.

It receives OTLP from producers, applies processors such as batching and memory limits, and exports onward to one or more backends.

In this design, the collector provides:

- protocol termination on localhost
- buffering and retry behavior
- a clean place for future filtering or enrichment
- a stable integration point if the backend changes later

### Langfuse

Langfuse is the backend and UI for inspecting agentic and LLM-oriented telemetry.

Langfuse documents native OpenTelemetry ingestion over HTTP on the `/api/public/otel` endpoint family and provides an OpenTelemetry-compatible way to receive data from instrumentation libraries and collectors.

In this design, Langfuse is the analysis surface, not the first integration boundary.

## Options evaluated

### Option A: Direct `Codex -> Langfuse`

Configure Codex to export OTEL directly to Langfuse without any collector.

This is the fewest-component design.

It is attractive because it looks simple, but it couples Codex directly to one backend and leaves less room for protocol adaptation, retries, filtering, or future fan-out.

It is also weaker in the face of the current Codex/Langfuse signal-shape mismatch.

### Option B: `Codex -> OpenTelemetry Collector -> Langfuse`

Configure Codex to export OTLP to a local collector, then configure the collector to export onward to Langfuse.

This adds one local component, but it is the most standard OTEL architecture and the most maintainable one.

It preserves optionality and gives a clean place to handle batching, memory limiting, retries, filtering, and future export to additional backends.

### Option C: No Codex OTEL, only app-level tracing

Skip Codex telemetry and only add observability to repository-specific applications or workflows later.

This avoids ambiguity around Codex signal shape, but it fails the immediate goal of making the current agentic workflow visible.

### Option D: Broader observability stack now

Adopt a bigger local stack immediately, such as collector plus multiple backends for logs, traces, and metrics.

This is operationally flexible, but it is too much complexity for the current phase.

## Evaluation criteria

Use these criteria, in order:

1. `Fit for current need`
   Does the option make the current Codex-based workflow observable now?
2. `Maintainability`
   Does the architecture stay easy to reason about for a small research-oriented repository?
3. `Cost effectiveness`
   Does the option help reduce wasted debugging time, wasted model usage, and repeated low-value agent behavior?
4. `Local setup complexity`
   Can one developer stand it up quickly on a laptop?
5. `Future flexibility`
   Does the choice keep room for filtering, fan-out, and additional instrumentation later?
6. `Backend coupling`
   Does the design avoid locking Codex directly to one observability backend?

## Decision

The repository should adopt `Codex -> OpenTelemetry Collector -> Langfuse` as the default local-development observability path, with local Docker Compose as the standard setup model.

This is the strongest option because it matches the immediate need for visibility into the Codex-based workflow while keeping the integration aligned with standard OpenTelemetry practice. It avoids coupling Codex directly to one backend, gives a clean place to absorb the current Codex-versus-Langfuse signal differences, and preserves a straightforward path for future filtering, fan-out, or richer instrumentation.

This decision is also about cost effectiveness, not only visibility. Better observability reduces wasted debugging time, makes retries and noisy execution paths visible, and helps identify where developer time and model usage are being lost. For this repository, agentic-development cost includes API spend, developer attention, investigation time, and repeated low-value work. The chosen architecture is intended to improve that full cost surface with evidence instead of guesswork.

## Target local architecture

The local target topology is:

```text
Codex
  -> OTLP/HTTP on localhost
  -> OpenTelemetry Collector Contrib
  -> OTLP/HTTP
  -> Langfuse
```

Codex should export to a local collector endpoint on localhost, and the collector should export onward to Langfuse’s OTLP HTTP endpoint under `/api/public/otel`. The first setup target should be Docker Compose because it is simple, familiar, and easy to inspect when something fails.

## Collector choice

The default collector distribution should be `otel/opentelemetry-collector-contrib`. It is the most practical open-source baseline for this repository because it is vendor-neutral, widely adopted, easy to run locally, and already includes the standard receivers, processors, and exporters needed for this design.

The first collector configuration should stay minimal: enable the `otlp` receiver, keep `batch` and `memory_limiter`, export with `otlphttp` to Langfuse, and leave filtering off by default. Bespoke proxy code or a thinner custom bridge should be avoided because they reduce standardization and create maintenance work without solving a real current problem.

## Configuration direction

The intended configuration direction is:

- Codex enables `[otel]`
- Codex uses an OTLP exporter rather than `none`
- Codex points to the local collector endpoint, not directly to Langfuse
- the collector owns backend-specific export configuration

The default initial Codex privacy posture should keep:

- `log_user_prompt = false`

The collector should export to the Langfuse OTLP HTTP endpoint using Langfuse’s documented authentication headers.

Because the current Codex documentation is written around OTEL event export and Langfuse’s OTEL documentation is trace-oriented, implementation should validate the exact signal path during setup instead of assuming semantic parity.

That is another reason the collector is part of the default design.

## Privacy defaults

Privacy defaults should stay conservative in the first iteration.

Use these defaults:

- do not export raw user prompt content by default
- do not add extra baggage or custom metadata propagation unless there is a concrete need
- keep secrets in environment variables, not in tracked files
- treat higher-fidelity prompt capture as an explicit opt-in later

The default local goal is operational visibility, not exhaustive content capture.

## Latency and operational tradeoffs

This design adds one local network hop.

That does introduce some overhead, but the expected tradeoff is favorable because:

- OTEL exporters batch asynchronously
- the collector batches and retries centrally
- local loopback traffic is low latency
- the collector reduces direct backend coupling and failure blast radius

In practice, the main tradeoffs are:

- direct export is simpler on paper
- collector-based export is more resilient and easier to evolve
- the collector makes failures easier to inspect because receipt, processing, and export can be debugged independently
- batching can slightly delay visibility, but it usually improves interactive stability rather than hurting it

The preferred bias is maintainability and inspectability over theoretical minimum component count. That same bias supports cost-effective development: when failures, retries, or slow paths are visible, the team can improve prompt design, tool choice, and workflow structure based on evidence instead of repeatedly tuning agent settings without understanding the real inefficiencies.

## Acceptance checks

The eventual implementation should satisfy these checks:

1. Codex starts with `[otel].exporter` enabled and points to a local OTLP HTTP collector endpoint.
2. The collector receives OTLP traffic from Codex.
3. The collector successfully exports to Langfuse’s OTLP HTTP endpoint.
4. A short Codex session produces visible telemetry in Langfuse.
5. With Langfuse unavailable, collector behavior remains inspectable and Codex does not require repository changes to recover.
6. With `log_user_prompt = false`, prompt content is not exported in the default setup.

## Rejected approaches

### Direct backend coupling from Codex

Rejected because it is the least flexible option and gives no clean integration boundary for protocol handling, retries, or future fan-out.

### No Codex observability for now

Rejected because it misses the immediate developer need: visibility into the current agentic workflow.

### Bigger observability platform now

Rejected because it adds complexity before the repository has a demonstrated need for multiple coordinated backends or production observability requirements.

### Bespoke proxy or custom bridge

Rejected because standard OTEL infrastructure already solves the current problem more cleanly and with less maintenance risk.
