## Summary

This document records a rejected local observability experiment.

The repository implemented a local stack around:

```text
Codex
  -> OpenTelemetry Collector
  -> self-hosted Langfuse
```

That experiment was useful as a technical probe, but it is not the current repository direction.

As of `2026-03-22`, this setup does not deliver the developer experience required for routine agent work in this repository.

## What Was Tried

The experiment attempted to use a local OpenTelemetry Collector plus self-hosted Langfuse to observe agent runtime activity without changing repository application code.

The desired outcome was a practical daily workflow that would make it easy to inspect:

- what the agent is doing
- where latency is accumulating
- token usage
- cost usage
- tool calls and tool results
- failures and retries

The repository also validated that Codex OTEL export can be enabled locally and routed through a collector into Langfuse.

## Why It Was Rejected

The experiment is rejected because the current state of the tooling, on `2026-03-22`, does not meet the actual product need.

The missing outcomes are:

- a good UI and UX for understanding what agents are doing end to end
- a reliable workflow for tracking latency, token usage, and cost in one place
- enough practical value to justify the setup and maintenance overhead

The stack can ingest telemetry, but successful ingestion alone is not the goal.

The goal is useful day-to-day visibility into agent behavior, performance, and cost. This experiment did not reach that bar.

## Codex-Specific Feedback

The repository was able to turn on OTEL export for Codex and confirm that the pipeline can work.

That said, the experience still falls short for normal use:

- the VS Code IDE extension introduces practical limitations around configuration and runtime behavior
- the resulting trace view still does not provide the quality of agent workflow visibility that is needed
- the setup does not give a satisfying combined view of latency, tokens, and cost for real development work

So the outcome is not that Codex observability is impossible.

The outcome is that the current Codex plus collector plus Langfuse path is not good enough for this repository right now.

## Repository Direction

The repository is not adopting this observability stack as an active supported workflow.

The previous implementation is kept only as archived reference material because it may still be useful later for:

- revisiting Codex observability when the IDE integration improves
- comparing Claude or other agent runtimes against the same OTEL boundary
- reusing some of the local collector and compose wiring in a future attempt

## Archive

The archived experiment assets now live under:

- `docs/design_docs/BNDD-0004/observability/`

This includes the previous local compose stack and related configuration.

These files are historical reference material, not current operational guidance.

## Revisit Conditions

This direction can be revisited later if one or more of these change:

- Codex or another agent runtime offers substantially better native observability UX
- latency, token, and cost data become easier to inspect in a single usable interface
- Claude or another runtime offers a better practical fit for the same operational goals
- the local setup cost becomes low enough relative to the information gained

Until then, the repository should treat this work as a useful experiment that did not justify adoption.
