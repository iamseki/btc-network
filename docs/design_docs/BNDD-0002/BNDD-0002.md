## Purpose

This benchmark remains useful historical evidence, but it no longer sets the repository default storage direction. `BNDD-0007` supersedes this benchmark-driven ClickHouse default with PostgreSQL for the current project phase.

This document defines a proof-of-concept benchmark for deciding whether `btc-network` crawler persistence should start on TimescaleDB or ClickHouse.

The goal is not to pick the fastest database in the abstract. The goal is to test which database is the better fit for the workload described in [BNDD-0001](/home/chseki/projects/personal/btc-network/docs/design_docs/BNDD-0001/BNDD-0001.md): append-heavy node observations, historical analytics, and a small but important "latest known state" view.

## Decision framing

The future crawler data model has two distinct shapes:

- `raw observations`: append-only events over time
- `current state`: the newest known view per endpoint

That split matters because the two database candidates are optimized differently.

### TimescaleDB fits best when

- the first implementation should stay close to PostgreSQL semantics
- `current state` queries, upserts, and relational joins matter as much as wide historical scans
- operator familiarity with Postgres is a practical advantage
- the expected dataset is large, but not yet so large that analytical scan speed dominates every other concern

### ClickHouse fits best when

- historical analytics is the primary product surface
- ingestion is mostly append-only and correction/update workflows are rare
- scan-heavy aggregations over very large fact tables dominate the workload
- denormalization and eventual consistency in derived views are acceptable tradeoffs

Both TimescaleDB and ClickHouse are open-source technologies with straightforward self-hosted deployment paths. That matters for this project because the storage layer should remain operable without dependence on a managed vendor platform.

## Why this benchmark exists

The crawler is not producing wallet-style transactional data. It produces repeated observations of network endpoints.

That workload usually stresses:

- sustained append throughput
- time-window aggregations
- top-k concentration queries by ASN, country, and prefix
- churn queries over recent windows
- latest-observation lookups per endpoint

The last item is the main source of tension:

- TimescaleDB handles it naturally with normal SQL patterns and relational modeling
- ClickHouse can do it, but the cleanest path usually relies on aggregate-state patterns, `argMax`, or derived tables/materialized views

## Benchmark goals

The benchmark should answer these questions:

1. Which engine loads synthetic crawler observations faster for the same dataset size?
2. Which engine answers the analytical queries faster?
3. How painful is the "latest state per endpoint" query in each engine?
4. How much operational complexity does each engine add for this project phase?
5. Does one engine require schema patterns that are clearly misaligned with the project's maintainability bias?

For TimescaleDB, the benchmark should evaluate compressed columnstore behavior, not only rowstore behavior, because historical analytics over long retention is part of the product direction.

## Partitioning default

The benchmark should use `monthly` partitioning/chunking as the default production-oriented layout.

Reason:

- the product is intended to run for years
- there is no historical backlog yet, but there will eventually be long-lived retained data
- monthly partitions age better operationally than daily partitions for moderate ingestion volumes

`Daily` partitioning is still useful as a comparison benchmark because some of the target queries are short recent windows. It should be treated as an alternate scenario, not the default assumption.

The benchmark kit therefore includes both schema variants and selects between them with `PARTITION_GRANULARITY=daily|monthly`.

The Docker benchmark environment currently defaults to TimescaleDB on PostgreSQL 18 so the evaluation reflects a modern self-hosted baseline rather than an older Postgres major.

The TimescaleDB benchmark path enables hypercore columnstore and manually converts chunks after load so storage and query measurements include TimescaleDB compression behavior.
It also runs `VACUUM FULL` and `ANALYZE` after conversion so post-load query and size measurements are taken from a compacted, statistics-refreshed state.

## Workload model

The synthetic data models the observation shape proposed in `BNDD-0001`.

Each row represents one crawler observation and includes:

- `observed_at`
- `crawl_run_id`
- `endpoint`
- `network_type`
- `handshake_status`
- `confidence_level`
- `protocol_version`
- `services`
- `user_agent`
- `start_height`
- `relay`
- `asn`
- `country_code`
- `prefix`
- `latency_ms`
- `discovered_count`

`discovered_count` is intentionally modeled as a per-observation interaction result, not as stable node state. It represents how many peer addresses a node returned during that specific discovery attempt.

The generator intentionally repeats endpoints across time so the benchmark exercises both:

- historical scans
- "latest row per endpoint" behavior

## Query suite

The benchmark runs the same logical workload against both engines:

1. Hourly handshake counts by network type over the last 7 days
2. Top 10 ASNs by verified reachable nodes over the last 24 hours
3. Version adoption by latest observation per endpoint
4. Daily churn: endpoints seen for the first time per day
5. Reachable-to-rumored ratio by day
6. Prefix concentration over the last 7 days
7. Latest-state snapshot grouped by handshake status

These queries are a better proxy for this project than generic TPC-style benchmarks.

## Synthetic data assumptions

The generator uses a fixed deterministic distribution so repeated runs are comparable.

The fake dataset approximates these patterns:

- most endpoints are `ipv4`
- a smaller slice is `ipv6`
- Tor/I2P-style overlay addresses exist but stay a minority
- most observations are successful handshakes
- some endpoints are only gossiped and never verified
- a smaller slice fails or times out
- user agents and protocol versions follow a skewed distribution
- ASN and country are low-cardinality dimensions

This is intentionally simple. The point is database fit, not perfect simulation of mainnet reality.

## Evaluation criteria

Use the benchmark results with these criteria, in order:

1. `Workload fit`
   Does the database match append-heavy observations plus latest-state queries without awkward patterns?
2. `Maintainability`
   Does the schema/query model stay easy to reason about for a small research project?
3. `Operational cost`
   How much extra tuning, view maintenance, and engine-specific behavior is required?
4. `Performance`
   Are ingestion and key analytics comfortably fast at the expected first-phase scale?
5. `Scale headroom`
   If the dataset grows 10x or 100x, does the chosen engine still look reasonable?

## Benchmark results

The current benchmark runs point in a clear direction: ClickHouse is outperforming TimescaleDB for this workload, and the gap grows as the dataset gets larger.

All runs below used `monthly` partitioning/chunking. The TimescaleDB figures include the explicit post-load steps used in this benchmark path: columnstore conversion, `VACUUM FULL`, and `ANALYZE`.

| Run | Dataset shape | TimescaleDB load | TimescaleDB query phase | TimescaleDB size | ClickHouse load | ClickHouse query phase | ClickHouse size |
| --- | --- | --- | --- | --- | --- | --- | --- |
| `small` | `250k obs / 25k endpoints / 14d` | `1860 ms` | `775 ms` | `18 MB` | `198 ms` | `154 ms` | `6.09 MiB` |
| `medium` | `1M obs / 100k endpoints / 30d` | `7518 ms` | `3153 ms` | `45 MB` | `578 ms` | `277 ms` | `25.32 MiB` |
| `large` | `5M obs / 500k endpoints / 60d` | `43434 ms` | `17646 ms` | `181 MB` | `2050 ms` | `547 ms` | `112.47 MiB` |
| `medium-high-cardinality` | `2M obs / 150k endpoints / 30d` | `15628 ms` | `5837 ms` | `78 MB` | `925 ms` | `313 ms` | `50.01 MiB` |

### Observations

- ClickHouse load time is consistently much lower than TimescaleDB load time.
- ClickHouse query time is consistently much lower than TimescaleDB query time, including the latest-state-style aggregate queries in this benchmark.
- ClickHouse storage is smaller in every run, even after TimescaleDB columnstore conversion and manual post-load maintenance.
- As observation volume and endpoint cardinality increase, the performance gap widens rather than narrowing.
- TimescaleDB remains workable at these scales, but it is not winning on the metrics that matter most for this observability-oriented workload.

## Decision

Historical note:

- this benchmark favored ClickHouse for the synthetic workload tested on `2026-03-16`
- the repository no longer treats that result as the canonical default for new work
- `BNDD-0007` changes the default backend choice to plain PostgreSQL for maintainability, operator familiarity, and lower self-hosted cost in the current experimental stage

The current benchmark evidence favors **ClickHouse** for this project phase.

Reason:

- the target workload is append-heavy and analytics-first
- eventual consistency for derived "latest state" views is acceptable for this use case
- benchmarked ingestion, analytical query performance, and storage efficiency all favor ClickHouse
- the gap remains visible even after giving TimescaleDB a more favorable compressed/maintained post-load path

That does not mean TimescaleDB is a bad fit in general. It means that for the workload described in `BNDD-0001`, and with the current acceptance that eventual consistency is fine, ClickHouse is the stronger default starting point.

Keep two cautions in mind:

- the synthetic dataset generator should still be tightened so both engines produce exactly the same logical rows
- a future implementation may still want a separate mutable/current-state representation if product requirements move away from eventual consistency
