## Purpose

This document narrows the durable-state shape for crawler restart recovery.

It supersedes the checkpoint-payload guidance in [BNDD-0005](../BNDD-0005/BNDD-0005.md) for implementation work on current branches. `BNDD-0005` still matters for the shared crawler/repository boundary, but it no longer describes the right on-disk recovery payload.

## Decision Summary

The repository will adopt this recovery design:

- keep `crawler_run_checkpoints` as append-only progress history only
- remove `resume_state` from checkpoints
- store durable recovery state in a separate append-only `crawler_run_recovery_points` table
- persist only the resumable workset as durable recovery payload
- define the resumable workset as the endpoints not fully completed at the recovery-point boundary, including queued work and work that was in progress
- restore all resumable-workset entries as pending work after a crash
- rebuild dedupe state from persisted `node_observations` for the run plus the restored resumable workset
- remove `node_states` from durable recovery state and from in-memory crawl state
- keep `discovered_node_states` as a counter only
- store the resumable-workset payload as versioned compressed bytes with encoding `zstd-json-v1`
- decouple stop-policy polling from durable checkpoint cadence with a dedicated `checkpoint_interval`

## Why This Decision Exists

The previous design stored a full `resume_state` snapshot in every checkpoint row.

That was a bad design decision for scale.

It mixed two different concerns into one write path:

- progress history for operators and analytics
- machine-oriented recovery state for crash restart

Because the crawler rewrote the full snapshot on every checkpoint, storage growth tracked the full accumulated in-memory state instead of the minimum state needed to resume. Large `seen_nodes`, `pending_nodes`, `in_flight_nodes`, and `node_states` sets were serialized repeatedly even though most of that data was either reconstructible or not required for restart.

The practical consequences were:

- checkpoint tables became dominated by one large blob field
- storage cost scaled with run duration and tracked-history size
- short checkpoint intervals amplified the problem
- analytics and run-list queries were forced to coexist with heavy recovery blobs

This decision fixes the data shape first and uses compression only on the remaining minimal payload.

## Design Requirements

The recovery design must satisfy these requirements:

- preserve startup recovery for the latest active run
- keep restart semantics explicitly at-least-once
- avoid persisting data that can be rebuilt from durable observations
- keep progress-history queries free of large opaque payloads
- keep the shared crawler workflow in `crates/btc-network`
- keep PostgreSQL-specific SQL and migrations in `crates/btc-network-postgres`

## Storage Model

The storage split is:

- `crawler_run_checkpoints`: append-only progress history for UI, API, and operator inspection
- `crawler_run_recovery_points`: append-only durable recovery snapshots used only for restart recovery
- `node_observations`: append-only observation history used both for analytics and for rebuilding dedupe state during recovery

`crawler_run_checkpoints` must not store recovery blobs.

`crawler_run_recovery_points` stores:

- run identity and phase metadata
- the same scalar metrics used for progress reporting
- `payload_encoding`
- `frontier_payload`
- `frontier_size`

In the current schema, `frontier_payload` and `frontier_size` hold the resumable workset.

## Recovery Semantics

The durable recovery payload contains only the resumable workset.

Recovery rules:

- queued endpoints and in-progress endpoints are persisted together as one resumable workset
- recovered in-progress work is re-queued as pending work
- `seen_nodes` is rebuilt from persisted observations for the same run plus the restored resumable workset
- repeated observations after crash recovery remain acceptable
- work after the last durable recovery point may be lost
- work after the last durable recovery point may also be replayed

`node_states` is not part of recovery anymore. It is not required to continue crawling, and keeping it in durable state was pure storage amplification.

## Compression Format

The recovery payload format is:

- encoding name: `zstd-json-v1`
- content: the sorted resumable workset only
- representation: versioned compressed JSON bytes

Compression is required, but only after the payload has been reduced to the minimal restart set.

## Configuration and Scale

Three knobs matter most for recovery payload behavior:

- `checkpoint_interval`: controls how often durable recovery points are written
- `max_tracked_nodes`: caps how large the tracked resumable workset can grow
- `max_concurrency`: affects crawl pressure and workset churn, but should not by itself dominate storage

Guidance:

- high concurrency is not inherently bad; it only became a storage problem because the old design rewrote the full accumulated state too often
- large `max_tracked_nodes` values still increase recovery payload size because the resumable workset can legitimately be larger
- a shorter `checkpoint_interval` improves crash freshness but increases write volume

Under this design, durable payload growth should track current resumable-workset size, not total visited-node history.

## Consequences

Benefits:

- checkpoint history becomes cheap to query and cheap to keep
- recovery payload size is bounded by resumable-workset size rather than total crawl history
- the crawler keeps restart support without persisting redundant state
- operator storage troubleshooting becomes clearer because progress history and recovery state are separated

Tradeoffs:

- crash recovery now depends on both the latest recovery point and persisted observations for the run
- recovery remains at-least-once rather than exactly-once
- resumable-workset payloads can still be large when the live outstanding work is large

## Verification

- `cargo test -p btc-network`
- `cargo test -p btc-network-postgres`
- `cargo test -p btc-network-crawler`

Required scenarios:

- startup recovery restores the latest active run from a recovery point
- corrupt recovery payloads are converted into terminal failed checkpoints before a new run starts
- recovery rebuilds dedupe state from persisted observations plus the restored resumable workset
- periodic durable writes use `checkpoint_interval`, not `lifecycle_tick`
- PostgreSQL migrations create separate checkpoint and recovery-point tables
