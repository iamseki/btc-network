## Purpose

This document defines how the crawler should handle outbound connection pressure when runtime networking limits are lower than crawler work demand.

The crawler is intentionally high-throughput, but raw worker concurrency is not the same thing as safe connect concurrency. Home networks, NAT tables, container file-descriptor limits, kernel networking limits, and cloud environments can all fail differently under the same crawler settings.

This BNDD makes crawler networking pressure an explicit design concern instead of an incidental side effect of `max_concurrency`.

## Decision Summary

The repository will adopt this runtime direction:

- separate outbound connect admission from overall worker concurrency
- keep connect pressure explicitly bounded by a dedicated crawler setting such as `max_in_flight_connects`
- treat worker concurrency, connect concurrency, retry budget, and connect timeout as different control surfaces with different cost drivers
- prefer delayed retry requeueing over immediate in-worker retry loops as the fairness-oriented scheduling model once bounded connect admission and telemetry are in place
- keep PostgreSQL pool sizing modest relative to crawler networking pressure because the crawler has one dedicated observation-writer task and one dedicated checkpoint-emitter task rather than a large set of concurrent database writers
- expose runtime pressure telemetry in periodic crawler progress logs so operators can distinguish peer-network pressure from database pressure
- keep runtime defaults pragmatic for local and common container environments while still allowing higher operator-tuned values in stronger environments
- prefer connect-phase backpressure and diagnostics over assuming every environment can safely absorb very large half-open connection counts

## Why This Decision Exists

The crawler recently produced runtime evidence that the dominant failure mode was outbound connect saturation rather than PostgreSQL saturation.

Observed progress summary:

- `in_flight_work = 5251`
- `open_fd_count = 4698`
- `tcp_established = 2`
- `tcp_syn_sent = 4686`
- `postgres_pool_size = 2`
- `postgres_pool_acquired = 0`

This is strong evidence that the crawler exhausted network-path or file-descriptor capacity with thousands of outstanding connect attempts while PostgreSQL remained mostly idle.

The important nuance is that `open_fd_count = 4698` did not mean the crawler had already hit the raised process `nofile` limit in that environment. The metric still mattered because it showed the crawler had already materialized thousands of live kernel objects, overwhelmingly sockets, and had therefore already pushed far beyond the scale that many local routers, NAT tables, or conntrack budgets tolerate comfortably.

The important design lesson is:

- high crawler throughput is desirable
- but uncontrolled connect bursts are environment-sensitive and can collapse consumer routers, NAT tables, conntrack limits, or local host networking before they create visible CPU, memory, or database pressure

## Observed Failure Mode

The current crawler architecture ties connect pressure too closely to worker concurrency.

Today a worker can:

1. dequeue an endpoint
2. attempt TCP connect
3. retry failed connects with backoff
4. continue occupying an in-flight work slot while the connect path is unresolved

At high concurrency, this allows thousands of simultaneous outbound connect attempts or retries.

Typical symptoms:

- local internet connectivity degrades or drops
- the crawler shows very high `tcp_syn_sent`
- file descriptors climb toward environment limits
- checkpoint or observation writes fail secondarily with `Too many open files`
- PostgreSQL metrics stay low, making the database error a symptom rather than the root cause

The write-path wording here matters:

- node observations are persisted by one dedicated writer task
- periodic checkpoints and recovery points are persisted by one dedicated checkpoint-emitter task
- there is not a large fan-out of concurrent observation writers

That is why low PostgreSQL pool activity in the observed run is meaningful evidence against the database being the dominant bottleneck.

## Metric Interpretation

The current progress metrics should be interpreted as follows.

### `open_fd_count`

This is a count of entries under `/proc/self/fd`, which the Linux manual describes as one entry for each file descriptor the process has open. That includes regular files, sockets, pipes, standard streams, epoll descriptors, and similar kernel objects.

For crawler diagnosis, the important point is not only "are we close to the configured `nofile` limit?" but also "what kind of runtime shape have we already created?" Thousands of open file descriptors in this crawler usually mean thousands of outstanding sockets plus a small number of runtime and I/O descriptors.

In the observed run, `open_fd_count = 4698` was still below the raised process limit. The metric still mattered because it showed the crawler had already created thousands of live sockets, enough to overload weaker network paths even before local descriptor exhaustion.

### `tcp_established`

This metric is derived from `/proc/self/net/tcp` and `/proc/self/net/tcp6`. The kernel documentation exposes the connection state in the `st` column; state `01` means `ESTABLISHED`.

`tcp_established = 2` means almost none of the crawler's sockets had completed the TCP handshake and become usable live peer sessions.

### `tcp_syn_sent`

This metric is also derived from `/proc/self/net/tcp` and `/proc/self/net/tcp6`. State `02` means `SYN-SENT`: the process has sent a SYN and is waiting for the peer side or network path to complete the handshake.

`tcp_syn_sent = 4686` means the crawler had thousands of outbound connections stuck in the initial connect phase. That is the strongest single signal that the runtime failure mode was connect-path saturation, not PostgreSQL saturation.

### Relationship Between These Metrics

The observed combination:

- `open_fd_count = 4698`
- `tcp_established = 2`
- `tcp_syn_sent = 4686`
- `postgres_pool_size = 2`
- `postgres_pool_acquired = 0`

means:

- almost every interesting file descriptor was tied up in outbound connect attempts
- almost none of those attempts had matured into active peer sessions
- the database pool was effectively idle

That is why this BNDD treats the dominant risk as outbound connection pressure rather than database concurrency.

## Design Requirements

The crawler runtime must satisfy these requirements:

- preserve high eventual crawl throughput in capable environments
- avoid coupling safe connect pressure to total worker count
- make the dominant runtime pressure visible in logs without requiring external tooling
- keep shared crawler control logic in `crates/btc-network`
- keep database-adapter specifics behind repository boundaries
- keep environment-sensitive values configurable by operators
- prefer direct, inspectable control surfaces over hidden adaptive heuristics for the first implementation

## Decision Details

### 1. Introduce Dedicated Connect Admission Control

The crawler should add a dedicated connect-phase budget, expected shape:

- config field: `max_in_flight_connects`
- behavior: a worker must acquire a connect permit before starting TCP connect or retry logic
- scope: permit covers connect attempts only, not the full handshake and persistence lifecycle

This preserves high worker concurrency for downstream work while preventing the crawler from creating a burst of half-open sockets that overwhelms the environment.

### 2. Keep Database Pool Small By Default

The crawler currently has:

- one observation writer task
- one checkpoint emitter task

That does not justify a large PostgreSQL pool by default. A modest pool leaves more file-descriptor budget and operational clarity for the network side, which is the dominant pressure source in current evidence.

### 3. Delayed Retry Requeueing Is The Preferred Fairness Model

When an endpoint fails to connect, the preferred fairness-oriented model is to requeue it for later consideration rather than retrying immediately inside the same worker loop.

Pragmatic direction:

- keep minimal retry metadata per endpoint
- start with `attempt_count`, `last_attempt_at`, and `last_failure_classification`
- derive retry eligibility from that metadata instead of storing duplicate scheduling state such as `next_eligible_at`
- reinsert retryable endpoints into the ready-work flow after a backoff decision rather than retrying inline inside the worker
- keep delayed retries owned by crawler runtime task management rather than detached fire-and-forget tasks

This keeps the state small and avoids duplicated timing truth while still reducing burst amplification against bad paths.

Possible advantages:

- immediate in-worker retries hold resources on the same bad endpoint
- delayed requeueing improves breadth-first sampling of the frontier
- delayed requeueing reduces repeated connect bursts against structurally bad peers
- retry ceilings remain explicit and easy to reason about

Tradeoffs:

- if the frontier has already collapsed to mostly unreliable nodes, delayed requeueing can converge toward behavior that is similar to immediate retry with backoff
- the scheduler and retry state become more complex
- the gain may be smaller than the complexity cost if bounded connect admission already solves most of the observed pressure problem

For that reason, this BNDD sequences delayed retry requeueing after the initial admission-control and telemetry work rather than putting it in the first implementation slice. It remains an important planned improvement because peer unreliability is a normal property of the network, immediate retries over-weight the same bad endpoints, and better fairness across the frontier should improve sampling quality while reducing unnecessary retry pressure.

### 4. Expand Runtime Pressure Telemetry

Periodic progress logs should expose enough signal to answer:

- are we bottlenecked on peer connects?
- are we exhausting file descriptors?
- are retries dominating throughput?
- is PostgreSQL pool pressure relevant?

Current progress telemetry now includes:

- `open_fd_count`
- `tcp_established`
- `tcp_syn_sent`
- `tcp_time_wait`
- `connect_slots_in_use`
- `connectable_tasks_started`
- `connect_retries_started`
- connect failure counters grouped by common error kind
- PostgreSQL pool size and idle/acquired counts

Future additions may still include:

- `dns_resolution_failures` when DNS resolution remains part of the runtime path

### 5. Keep Defaults Environment-Aware, Not Environment-Dependent

The shared crawler defaults should remain reasonable for common local or container execution.

The Compose/local operator path may still expose higher defaults, but only when:

- open-file limits are raised explicitly
- connect admission control is active
- documentation names the tradeoff clearly

For local laptop-oriented development, the repository defaults should remain conservative. In this rollout the Compose crawler service uses `ulimits.nofile=1024` by default because that is a common host-shell baseline and because a very high container limit is a misleading local default when router or upstream NAT limits usually fail first.

This BNDD does not require an auto-tuning controller in the first slice. Explicit knobs plus telemetry are the preferred first step.

## Peer Reliability Assumptions

The crawler must treat peer unreliability as a normal property of Bitcoin P2P crawling rather than as exceptional behavior.

Important assumptions:

- some candidate endpoints will not accept new inbound TCP connections
- some nodes will drop or time out during the Bitcoin handshake
- some nodes will accept a connection but provide little or no useful peer-discovery data
- some advertised addresses will be stale, unreachable, rate-limited, firewalled, or behind changing network paths
- low handshake success rates are expected in broad internet crawling and do not by themselves imply a crawler defect

This matters for runtime control because many failures are structural or transient in the peer population itself. The crawler should therefore optimize for bounded pressure and useful sampling rather than assuming immediate retries can force most candidates into successful sessions.

## Operational Guidance

Operators should interpret the runtime control surfaces as follows:

- `max_concurrency`: total work parallelism budget
- `max_in_flight_connects`: maximum active outbound connect pressure
- `connect_timeout`: how long one connect attempt may occupy a connect slot
- `connect_max_attempts`: how much retry amplification the crawler may create
- delayed retry requeueing: a built-in way to spread retry pressure over time instead of hammering the same bad endpoint immediately
- `connectable_tasks_started`: count of connect-eligible endpoints that workers started processing; useful for throughput context, but not identical to raw TCP connect syscall count
- `postgres_max_connections`: modest write-side database pool budget, not a throughput lever for peer discovery

Expected troubleshooting interpretation:

- high `tcp_syn_sent` + low PostgreSQL acquisition means network-path pressure
- high `writer_backlog` + high PostgreSQL acquisition means persistence-side pressure
- high `tcp_time_wait` means connect churn is too bursty or retries are too aggressive
- `open_fd_count` near process limit means the crawler is close to local resource exhaustion even if CPU and memory are healthy

### Inspect Local Limits

The phrase "router, NAT table, or conntrack budget" is environment-specific. The repository should document how to inspect the host and container limits that most directly shape crawler behavior on Linux.

Useful checks:

```bash
# host shell open-file limit
ulimit -n

# container open-file limit
docker exec btc-network-crawler sh -lc 'ulimit -n'

# Linux conntrack table capacity
sysctl -n net.netfilter.nf_conntrack_max

# host ephemeral port range available for outbound connections
sysctl -n net.ipv4.ip_local_port_range

# kernel TCP SYN retry budget
sysctl -n net.ipv4.tcp_syn_retries

# optional, if conntrack tooling is installed
conntrack -S
```

Interpretation:

- `ulimit -n` bounds how many file descriptors the crawler process may hold open at once
- `nf_conntrack_max` is the kernel tracking-table capacity for network flows on the host, but it is not the same thing as the effective NAT capacity of a home router upstream from the host
- `ip_local_port_range` bounds the local ephemeral port space available for outbound sockets from the host namespace
- `tcp_syn_retries` influences how long failed connect attempts can remain in-flight before the kernel gives up
- `conntrack -S` is useful when available because it shows live conntrack counters rather than only the configured ceiling
- local laptop shells often still default to `ulimit -n = 1024`, so very large container `nofile` values should be treated as explicit operator overrides rather than assumed-safe local defaults

These values should be used as operator context, not as a guarantee that the upstream router or ISP path can sustain the same scale. Consumer network gear can fail well before the Linux host reaches its own configured conntrack or file-descriptor ceilings.

## Consequences

Benefits:

- crawler can stay high-throughput without assuming every environment tolerates massive connect bursts
- runtime failures become easier to attribute correctly
- database troubleshooting becomes less noisy because database errors are less likely to be secondary symptoms of peer-network saturation
- operators get explicit knobs that map to real cost drivers

Tradeoffs:

- one more crawler config knob adds operational complexity
- connect admission can reduce short-term frontier drain speed in very strong environments if configured too conservatively
- delayed retry requeueing may add scheduler complexity without large benefit when the remaining frontier is already dominated by unreliable peers
- telemetry adds implementation and maintenance cost in the shared runtime path

## Verification

- `cargo test -p btc-network`
- `cargo test -p btc-network-crawler`
- `cargo test -p btc-network-postgres`

Required scenarios:

- crawler enforces a separate cap for active connect attempts
- workers continue making progress when handshake or persistence work remains but connect permits are exhausted
- runtime progress logs include connect-pressure and pool-pressure fields
- high worker concurrency with a lower connect budget no longer produces unbounded `SYN-SENT` growth from crawler control flow alone
- Compose and local docs describe how connect admission, timeouts, retries, and open-file limits interact
