## Purpose

This document records how the crawler should treat onion peers and Tor-backed connectivity.

The crawler already supported direct TCP reachability for IPv4 and IPv6 peers. Bitcoin peer discovery also advertises onion endpoints, but those endpoints do not map to stable public IP addresses and cannot be reached through the normal clearnet socket path.

The goal of this BNDD is to make onion reachability explicit without contaminating the existing clearnet crawler model.

## Decision Summary

The repository adopts this direction:

- keep Tor as an additional reachability layer rather than merging it into the direct IPv4 and IPv6 flow
- preserve onion peers from addrv2 peer discovery as first-class crawler endpoints
- attempt onion connections only through an explicitly configured SOCKS5 proxy backed by Tor
- keep Tor support optional so environments without a configured proxy preserve existing clearnet behavior
- keep persisted endpoint classification explicit through `network_type`
- prefer operator-owned Tor clients over third-party public proxy services

This decision is implemented in the repository.

## Why This Decision Exists

Without Tor support, the crawler under-observes a meaningful slice of the Bitcoin network.

The important distinction is that onion peers are not just another host string. They require Tor-mediated routing, different operational prerequisites, and a different failure surface from direct TCP.

Treating Tor as a separate reachability layer keeps the crawler model clearer:

- IPv4 and IPv6 remain direct socket work
- onion peers remain observable when Tor is available
- non-Tor environments do not fill the dataset with artificial onion failures

## Design Requirements

The crawler must satisfy these requirements:

- detect onion peers from normal peer discovery
- preserve existing direct-connect behavior for IPv4 and IPv6 peers
- require explicit Tor proxy configuration before onion peers are queued for follow-up attempts
- route onion connection attempts through SOCKS5
- keep the design simple and maintainable rather than adding a large Tor-specific control plane in the first slice
- document operator requirements clearly enough that Tor support is practical to run

## Decision Details

### 1. Onion Endpoints Stay In Discovery Output

When addrv2 discovery includes `TorV2` or `TorV3` addresses, the crawler keeps them as crawler endpoints with Tor-specific network metadata.

The crawler now converts those addresses into canonical `.onion` hostnames instead of storing raw hex as the dial target.

### 2. Persisted Classification Stays Simple

Persisted observations continue to carry `network_type`.

For the current implementation, that is enough to distinguish Tor observations from clearnet ones because the runtime invariant is still:

- `tor_v2` and `tor_v3` use the configured Tor SOCKS5 path
- `ipv4` and `ipv6` use direct TCP

If a future implementation allows the same endpoint class to be reached through multiple transport paths, the schema can be revisited then.

### 3. Tor Attempts Require Explicit Proxy Configuration

The crawler exposes an optional `tor_socks5_addr` runtime setting.

Behavior:

- if `tor_socks5_addr` is configured, discovered onion peers are eligible for follow-up scheduling
- if `tor_socks5_addr` is not configured, discovered onion peers are not queued for follow-up attempts
- if a Tor endpoint is processed directly without proxy configuration, the attempt fails at connect classification

This keeps the non-Tor default clean. The crawler does not fabricate onion reachability failures in environments that never intended to run Tor.

### 4. SOCKS5 Is The Boundary, Not Tor Control Integration

The first implementation slice stops at SOCKS5.

Non-goals for this slice:

- no Tor control-port integration
- no circuit rotation management
- no multi-proxy failover logic
- no separate Tor retry budget

Those may be revisited later if operational evidence justifies them.

## Operator Options

The crawler does not require a special public Tor proxy service. It requires a SOCKS5 endpoint backed by Tor that the operator controls or trusts.

Recommended default:

- run the repository's dedicated Tor client container on the same Docker
  network as the crawler
- point the crawler at `tor:9050`

Other acceptable options:

- a local `tor` daemon from the Tor Project or operating-system packages, commonly exposing `127.0.0.1:9050`
- Tor Browser's local SOCKS port, commonly exposing `127.0.0.1:9150`
- a self-hosted Tor client in a VM or separate host that exposes a trusted SOCKS5 endpoint

Important constraints:

- a generic public SOCKS5 proxy is not enough for `.onion` routing
- the recommended containerized path is an operator-owned Tor client on the same Compose network as the crawler
- the repository Compose path includes a Tor healthcheck that waits for bootstrap progress to reach `100%` through the local control port rather than only checking whether port `9050` is open
- if the crawler runs in a container, `127.0.0.1:9050` points at the crawler container itself rather than the Tor service
- if Tor runs in Docker but the crawler runs on the host, publish the SOCKS port only on loopback and keep the crawler target at `127.0.0.1:9050`
- Tor bootstrap latency is normal; even after a healthy transition there can still be a small gap before first usable onion traffic on slow networks
- the current crawler SOCKS path only supports unauthenticated SOCKS5
- Tor support should remain opt-in rather than silently relying on third-party infrastructure

Example crawler configuration:

```bash
btc-network-crawler --tor-socks5-addr 127.0.0.1:9050
```

Example Compose-oriented crawler configuration:

```yaml
services:
  tor:
    build:
      context: .
      dockerfile: apps/tor/Dockerfile
    healthcheck:
      test:
        [
          "CMD-SHELL",
          "printf 'AUTHENTICATE\\r\\nGETINFO status/bootstrap-phase\\r\\nQUIT\\r\\n' | nc 127.0.0.1 9051 | grep -q 'PROGRESS=100'",
        ]

  crawler:
    depends_on:
      tor:
        condition: service_healthy
    environment:
      BTC_NETWORK_CRAWLER_TOR_SOCKS5_ADDR: tor:9050
```

## Consequences

Positive consequences:

- onion peers become reachable in Tor-enabled environments
- the crawler data model stays simple and explicit about endpoint type
- direct IPv4 and IPv6 behavior remains unchanged
- operators can enable Tor without forcing every deployment to run it

Tradeoffs:

- Tor-disabled runs still discover onion peers but intentionally skip follow-up attempts
- Tor operational visibility currently comes mostly from persisted observations, failure classification, and logs rather than a large set of Tor-specific counters
- resilience remains limited to the existing retry model and one configured SOCKS5 endpoint

## Verification

- `cargo test -p btc-network`
- `cargo test -p btc-network-postgres`
- `cargo test --manifest-path apps/crawler/Cargo.toml`

Required implemented scenarios:

- onion addrv2 entries become crawler endpoints with `.onion` hostnames
- Tor endpoints are only queued for follow-up when a SOCKS5 proxy is configured
- successful and failed observations persist Tor endpoint classification through `network_type`
