## Purpose

The normal crawler should not spend its main network budget retrying endpoints that were recently known to be unreachable. Those endpoints still need a path back into the reachable set because Bitcoin peer failures can be temporary.

This BNDD defines a split flow:

- normal crawler excludes recently unreachable endpoints from normal connection attempts
- recovery crawler retries only unreachable endpoints and restores those that become reachable again

## Decision Summary

The repository uses a dedicated `unreachable_nodes` table plus a startup load query over recent active unreachable state.

Normal crawler behavior:

- load unreachable endpoints at startup
- store them in an in-memory `HashSet<String>` keyed by canonical endpoint
- skip those endpoints during seed enqueueing and discovered-peer enqueueing
- retry reachable endpoint failures up to the configured attempt budget, default `5`
- after all connect attempts fail, persist the terminal failed observation and record the endpoint in `unreachable_nodes`
- on successful visits, persist the normal `node_observations` row

Recovery crawler behavior:

- run the explicit `recover-unreachable` crawler subcommand
- load only unreachable endpoints
- retry reachable endpoint failures up to the same configured attempt budget, default `5`
- do not follow discovered peers
- when an endpoint succeeds, persist the observation and soft-delete unreachable state by setting `recovered_at`
- when an endpoint still fails after all retries, keep or refresh unreachable state

## Why This Decision Exists

Dominant cost driver is failed outbound peer connection work: socket creation, connect timeout latency, retry backoff, and terminal persistence for endpoints that a recent crawl already proved unreachable.

Skipping known unreachable endpoints reduces normal crawl wall time and outbound network pressure before workers spend connect permits. It also keeps the normal crawler focused on fresh discovery rather than repeatedly sampling stale bad paths.

## Decision Details

### In-Memory Membership

The normal crawler stores excluded endpoints in `HashSet<String>` keyed by canonical endpoint.

Reason:

- lookup is expected `O(1)`
- canonical endpoint is already stable across storage, discovery, and logging
- the skip decision does not need the full endpoint payload
- using a hash set avoids per-candidate database lookups in the hot enqueue path

### Startup Source

The PostgreSQL adapter loads exclusions from:

- active rows in `unreachable_nodes` whose latest unreachable timestamp is inside the configured lookback

This preserves compatibility with existing observation history while giving new runs a compact dedicated table for steady-state lookup.

### Retry And Write Path

Retryable reachable endpoint failures are requeued with existing delayed retry logic and are not persisted as terminal unreachable state until the attempt budget is exhausted.

Terminal reachable endpoint failures create:

- a failed observation row
- an `unreachable_nodes` upsert

Successful recovery visits create:

- a successful observation row
- a soft-delete update on `unreachable_nodes`

### Recovery Path

The recovery path is exposed as the crawler binary's `recover-unreachable` subcommand. It reuses shared crawler behavior but disables discovered-peer fanout so it processes only the unreachable set.

## Tradeoff: Derived State Versus Dedicated Table

Deriving unreachable state only from `node_observations` keeps writes append-only and avoids another table, but lookup cost grows with observation history. Every normal run would need a latest-row query across a high-cardinality table before it can skip known bad endpoints.

A dedicated `unreachable_nodes` table adds one small state write for terminal reachable endpoint failures and one soft-delete write for recovery success. In return, active unreachable lookups stay compact, indexable, and independent of append-only observation growth.

Chosen tradeoff:

- maintain `unreachable_nodes` as the operational state table
- avoid extra mutable fields such as `last_tried` until recovery scheduling needs them

## Consequences

- normal crawl path avoids recent known-bad endpoints before spending connect permits
- unreachable endpoints remain recoverable through an explicit recovery subcommand
- recovery cadence can be scheduled separately without making normal crawler control flow dirtier
- the dedicated table must be migrated and maintained by PostgreSQL adapters
