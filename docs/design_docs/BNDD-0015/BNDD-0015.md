## Purpose

The current API uses bounded `limit` parameters. That protects response size but does not always answer two different questions:

- can a client intentionally scan row-level data?
- can the backend avoid open-ended historical aggregation over growing tables?

This BNDD narrows pagination to endpoints that actually need continuation and uses date/run bounds for aggregate analytics. The project is pre-production, so the API may make breaking contract changes.

## Decision Summary

Adopt a mixed strategy:

- use keyset pagination only for row-level collections that clients may reasonably scan
- keep compact aggregate endpoints as bounded top-N responses
- add explicit date or run bounds to historical aggregate endpoints whose cost grows with `node_observations`
- do not paginate curated or low-cardinality endpoints
- do not implement offset pagination
- keep endpoint-specific max `limit` caps

This avoids turning every endpoint into a cursor API while still giving clients a safe scan path where it matters.

## Endpoint Classification

| Endpoint | Direction | Reason |
| --- | --- | --- |
| `GET /api/v1/network/last-run/nodes` | keyset pagination | row-level table over latest run; clients may need to scan verified nodes |
| `GET /api/v1/network/historical/runs` | optional keyset or date-bounded list | run count grows slowly, but date filtering is useful for operators |
| `GET /api/v1/network/historical/asns` | require date window or replace with run-scoped query | current all-history aggregate can grow with `node_observations`; pagination is not the right fix |
| `GET /api/v1/network/historical/runs/{run_id}` | no pagination in current shape | detail is summary plus compact latest checkpoints; add dedicated checkpoint endpoint only if full checkpoint scans become needed |
| last-run distribution endpoints | no pagination | compact top-N analytics; full scans are not a current product need |
| `GET /api/nodes/status` | no pagination | curated target set is intentionally small; history stays bounded by retention/window |
| docs routes | no pagination | static/generated docs |

Last-run distribution endpoints include services, protocol versions, user agents, network types, countries, ASNs, start heights, and ASN organizations.

## API Contract Direction

### Keyset Endpoints

Only row-level scan endpoints should accept:

```text
limit=<positive bounded integer>
pageToken=<opaque string>
```

They should return:

```json
{
  "items": [],
  "nextPageToken": "opaque-token-or-null"
}
```

Rules:

- `nextPageToken = null` means no more pages
- tokens are endpoint-scoped
- malformed or wrong-endpoint tokens return `400 bad_request`
- token internals are not API contract

### Date-Bounded Historical Endpoints

Historical aggregate endpoints should prefer explicit bounds:

```text
start=<RFC3339 timestamp>
end=<RFC3339 timestamp>
limit=<positive bounded integer>
```

Rules:

- enforce `start < end`
- enforce a maximum time window, such as `31 days`, unless a BNDD chooses another value
- reject missing bounds for historical aggregate endpoints that would otherwise scan all observations
- keep `limit` as top-N cap, not pagination

Run-scoped alternatives are acceptable when the data naturally belongs to one crawler snapshot.

## Storage Direction

Only add surrogate cursor keys where needed.

Likely first schema change:

- `node_observations.node_observation_id BIGSERIAL PRIMARY KEY`

Use it as the stable tie-breaker for `last-run/nodes` keyset pagination alongside `observed_at` and `endpoint`.

Do not add `BIGSERIAL` IDs to `crawler_run_checkpoints` unless a dedicated checkpoint scan endpoint is approved.

Existing `node_status.node_status_id` is enough for status history internals.

## Tradeoffs

Keyset pagination is valuable for row scans, but applying it to every aggregate endpoint adds contract and frontend complexity without clear product value.

Date/run bounds solve the more important aggregate risk: avoiding open-ended scans across growing observation history. This makes backend cost more predictable and easier for agents to reason about.

Leaving distributions as top-N responses means clients cannot enumerate every rare bucket today. That is acceptable because the current UI and agent guide need compact analytics, not exports.
