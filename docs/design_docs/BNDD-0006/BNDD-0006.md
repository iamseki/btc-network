## Purpose

This document settles the next product-facing slice after durable crawler persistence.

The goal is to expose persisted crawler analytics in the real web and desktop UI without:

- leaking storage-adapter details into React
- adding crawler analytics Tauri commands
- expanding into crawler control, auth, or browser-side mutation in the same slice

## Decision Summary

The repository will adopt this direction for crawler analytics UI:

- the next frontend slice is analytics-first, not crawler-control
- crawler analytics reads are public and read-only in this phase
- `apps/web` becomes the primary analytics UI
- `apps/desktop` remains a thin shell over the same React pages
- a new Rust HTTP app under `apps/api` is the browser-safe backend for crawler analytics reads
- the shared Rust layer owns the crawler analytics read contract and read models
- the PostgreSQL adapter implements the read contract behind shared traits
- the preserved ClickHouse adapter remains available only as a documented legacy path
- both web and desktop analytics reads use the same HTTP client helper and API base URL
- `Network Analytics` is the default analytics landing page in the current MVP shell
- analytics pages use a second-level header navigation so each surface can grow without overloading the global sidebar
- existing single-peer pages remain in the product as a separate section

## Implementation Plan

Execution tracking lives in [implementation-plan.md](./implementation-plan.md).

## Contract Scope

The read-only shared contract for this phase is:

- `listCrawlRuns`
- `getCrawlRun`
- `countNodesByAsn`

The app-facing shared models for this phase are:

- `CrawlRunListItem`
- `CrawlRunDetail`
- `CrawlRunCheckpointItem`
- `FailureClassificationCount`
- `NetworkOutcomeCount`
- `AsnNodeCountItem`

`CrawlRunListItem` exists to keep the UI thin. The API should return derived fields such as:

- `success_pct`
- `scheduled_pct`
- `unscheduled_gap`

so the frontend does not re-encode query semantics.

## Architecture Boundaries

The intended shape is:

```text
React page
  -> apps/web/src/lib/api client boundary
  -> shared HTTP analytics helper
  -> apps/api
  -> shared crawler analytics trait in crates/btc-network
  -> PostgreSQL adapter in crates/btc-network-postgres
  -> PostgreSQL
```

Rules:

- do not import storage-adapter crates in `apps/web`
- do not add crawler analytics Tauri commands in `apps/desktop`
- do not expose raw SQL or table names through the API
- keep query logic in Rust adapters, not in React components
- keep the desktop shell focused on native-only capabilities for the single-peer flows that still need them

## HTTP Surface

The minimal browser-safe HTTP surface for this phase is:

- `GET /api/v1/crawler/runs?limit=<n>`
- `GET /api/v1/crawler/runs/:run_id`
- `GET /api/v1/crawler/asn?limit=<n>`

Response rules:

- JSON only
- read-only
- bounded page sizes
- generic internal error messages that do not leak adapter-specific details

## Operational Hardening

The current HTTP app includes a small public-read hardening layer:

- allowlist-based CORS rather than wildcard CORS
- bounded request timeouts
- bounded in-flight request concurrency

Current limits of that hardening:

- the API is still intentionally public and read-only rather than authenticated
- there is not yet a dedicated rate-limiting or abuse-throttling layer beyond the concurrency cap
- production deploys should set `BTC_NETWORK_API_ALLOWED_ORIGINS` explicitly instead of relying on the broader local-dev-friendly defaults

These defaults are environment-configurable at deploy time through:

- `BTC_NETWORK_API_ALLOWED_ORIGINS`
- `BTC_NETWORK_API_REQUEST_TIMEOUT_SECS`
- `BTC_NETWORK_API_CONCURRENCY_LIMIT`

## Frontend Runtime Rules

The frontend adapter boundary under `apps/web/src/lib/api/` remains the only UI-facing client surface.

Runtime behavior:

- web runtime uses HTTP for crawler analytics by default
- hosted browser-only builds may opt into deterministic demo analytics data when the public API is not deployed yet
- hosted demo analytics may also replay a client-only latest-snapshot lifecycle from the most recent run detail so the public shell can feel active without crawler control or live backend polling
- desktop runtime also uses HTTP for crawler analytics
- desktop runtime continues to use Tauri commands for the existing single-peer flows
- plain web runtime still uses placeholder/mock responses for the single-peer flows that do not yet have a browser-safe backend

Configuration:

- `VITE_API_BASE_URL` points the frontend at the analytics API
- `VITE_DEMO_MODE` switches hosted browser builds to deterministic demo analytics data instead of HTTP reads
- local development defaults to `http://127.0.0.1:8080`
- production defaults to `https://api.btcnetwork.info`

## UI Shape

The existing shell and page registry stay in place.

This phase adds two top-level pages:

- `Crawler Runs`
- `Network Analytics`

Navigation is grouped into:

- `Network Analytics`
- `Peer Tools`

The intended page behavior is:

- `Crawler Runs`: compact recent-run picker, selected-run detail, recent checkpoints, failure mix, and network outcome mix
- `Network Analytics`: a score-first network-risk home page, latest-snapshot hero, top ASN counts, recent verification outcome breakdowns, and lightweight comparative charts where the current API already supports them

The current single-peer pages remain:

- `Connection`
- `Peer Tools`
- `Chain Height`
- `Block Explorer`

## Current Implemented Shape

The implemented shell stays within the existing page-registry structure and keeps analytics growth inside page-local secondary navigation rather than adding a router rewrite.

Current behavior on this branch:

- `Network Analytics` is the first landing page in the shell
- analytics pages keep a compact second-level header nav aligned with the selected page
- analytics pages keep their high-signal state in compact header-row stats next to refresh actions instead of duplicating that context in large summary grids
- analytics pages expose the latest crawler snapshot in the global header rail as a compact status chip that expands into a centered preview overlay and routes back into the `Network Analytics` home page
- hosted demo mode replays a deterministic five-minute latest-snapshot cycle, persists replay state in browser local storage, rests for fifteen minutes before replaying again, and starts a fresh live cycle when the user returns after a longer absence
- `Crawler Runs` is now a cleaner inspection surface only: recent runs, selected-run detail, checkpoints, failures, and network outcomes, with selected-run metrics compact in the heading row
- the recent-run selector is intentionally compact and phone-friendly so the crawler page avoids dense horizontal table chrome on narrow screens
- `Network Analytics` keeps focused slices for overview, top ASNs, and verification outcomes, uses lightweight inline charts for ASN concentration and verification distribution without adding a dedicated charting dependency, and derives score-first product signals such as decentralization, eclipse-exposure proxy, observation confidence, and transport diversity from the current read-only contract
- the `Network Analytics` overview now follows a clearer product-home hierarchy:
  - globe-first snapshot hero
  - compact `Risk Brief` as the secondary companion panel
  - `Risk Drivers` as a full-width evidence strip beneath the hero row
  - supporting charts after the hero and driver layers
- on small screens the snapshot hero remains first and `Risk Brief` follows second; on larger screens the hero stays left and `Risk Brief` stays right
- the public home no longer reuses the crawler-specific checkpoint rail; that rail remains only on crawler inspection surfaces where process internals still matter
- hosted browser deploys can preserve those analytics pages in a browser-only demo mode through deterministic embedded data when the HTTP API is not ready yet, including realistic crawler snapshot sizes instead of tiny placeholder counts
- the session log stays on the single-peer pages and is intentionally hidden on analytics pages to keep those surfaces cleaner

## Agent Notes

For future frontend work in this slice:

- treat `Network Analytics` overview as the product home page for public network-observability work
- keep the main story score-first and evidence-backed: snapshot and scores first, risk drivers next, charts and tables after that
- derive product signals from the current read-only contract rather than inventing hidden browser-only data sources
- keep `Crawler Runs` as an inspection page, not a second home page
- keep the public home globe-first; do not let right-rail score boxes or crawler-process telemetry displace the snapshot hero
- keep `Risk Brief` compact and secondary to the hero, especially on mobile
- keep `Risk Drivers` product-facing and explanatory; avoid reintroducing checkpoint rails or crawler-internal progression blocks on the public home
- prefer concise docs here over repeated repo-wide guidance; if the shipped UI shape changes, update this BNDD and the compact agent docs together

## Out of Scope

This phase does not include:

- `startCrawl`
- `stopCrawl`
- mutation endpoints
- desktop-only analytics commands
- browser-side direct storage-adapter access
- websockets, streaming UI updates, or background polling
- dashboard-heavy visualization libraries or map-style dependencies
- protocol-grade decentralization or eclipse-risk scoring backed by richer topology data than the current API exposes
- authentication or authorization for crawler control
- full rate limiting, authn/z, or abuse-prevention layers for the public read-only API beyond the current CORS, timeout, and concurrency guardrails

Those belong to a later control-plane slice if the product needs them.
