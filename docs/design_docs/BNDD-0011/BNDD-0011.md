## Purpose

This document proposes the first deployment strategy for the public Rust API and its PostgreSQL database.

The repository already has a production path for the static frontend:

- `apps/web` on Cloudflare Pages
- browser-safe analytics reads through `apps/api`
- PostgreSQL as the default analytics storage backend

What is still missing is the first concrete decision for:

- where the public API should run
- where PostgreSQL should run
- how the public API should be exposed safely
- how IP-based rate limiting and basic DDoS protection should work without committing early to an expensive managed stack

This BNDD narrows the future API phase from [BNDD-0003](../BNDD-0003/BNDD-0003.md) into a specific low-cost, self-managed direction.

## Decision Summary

If accepted, the repository should use this strategy:

- keep the static web frontend on `Cloudflare Pages`
- expose the public API only at `api.btcnetwork.info`
- move authoritative DNS for `btcnetwork.info` to `Cloudflare` when the API goes live, while keeping domain registration at Hostinger unless a registrar move becomes operationally useful
- place the Rust API behind a Cloudflare-proxied HTTPS edge
- run the Rust API on a self-managed VM with the service bound to loopback behind a reverse proxy
- keep PostgreSQL self-managed and non-public by default
- start with API and PostgreSQL on the same VM to minimize fixed cost and operational sprawl
- use `Oracle Cloud Infrastructure Always Free` as the zero-dollar bootstrap target when it is actually available
- use `Hetzner Cloud` as the first paid fallback and the default ongoing low-cost paid path
- do not treat `AWS EC2` as the default for this repository's first public API phase
- start rate limiting by source IP at the Cloudflare edge and enforce a second per-IP limit at the origin reverse proxy

## Why This Decision Exists

The user goals for this phase are specific:

- keep costs at or near `$0` if possible
- keep full server control
- avoid managed platform lock-in for the API and database
- make the public API safe enough for anonymous internet traffic
- keep the first deployment simple enough to operate alone

Those goals push the design toward one practical shape:

- static frontend at the edge
- one small self-managed VM
- one local PostgreSQL instance
- one public API hostname behind Cloudflare

The main question is not whether a more elaborate platform is possible.

It is whether a simpler shape can be made safe enough and cheap enough to ship first.

## Constraints

This decision must respect these constraints:

- the frontend deployment path already exists and should remain low-friction
- the API needs public internet access
- the database should not be public
- the project wants server control, not only managed runtimes
- rate limiting should start by IP before introducing API keys or tenant-specific policy
- DDoS posture should rely first on cheap edge protection, not on oversizing the VM
- the repository should avoid Kubernetes, service meshes, or multi-node clusters in the first deployment slice
- the project should not assume paid managed PostgreSQL

Two consequences follow:

1. The public edge and the origin server should be treated as separate concerns.
2. The cheapest acceptable design is one VM plus one edge provider, not a fleet.

## Dominant Cost Drivers

For this phase, the dominant cost drivers are:

1. always-on VM compute
2. always-on block storage for PostgreSQL
3. backup retention and off-host copies

Request volume is not the first cost driver because:

- the frontend stays static at the edge
- Cloudflare can absorb some abusive or bursty traffic before it reaches origin
- expected initial API traffic is modest

Bandwidth can become a later cost driver, but it should not drive the initial topology.

That means the correct first optimization is:

- minimize always-on origin footprint
- keep PostgreSQL on the same host until evidence says otherwise
- keep abusive traffic off origin as early as possible

## Options Evaluated

### Option A: Cloudflare edge plus OCI Always Free VM

Keep the static web path unchanged, proxy `api.btcnetwork.info` through Cloudflare, and run the Rust API plus PostgreSQL on an OCI Always Free ARM VM.

Why it matters:

- this is the only realistic zero-dollar always-on VM path among the evaluated options
- OCI Always Free includes up to `4 OCPUs`, `24 GB` memory, `200 GB` block storage, `10 TB` outbound transfer, and one small Always Free load balancer in the tenancy home region

Key risks:

- Always Free capacity can be unavailable in some regions
- idle instances can be reclaimed
- the free tier is attractive but less predictable than a paid VPS

### Option B: Cloudflare edge plus Hetzner Cloud VM

Keep the static web path unchanged, proxy `api.btcnetwork.info` through Cloudflare, and run the Rust API plus PostgreSQL on one Hetzner Cloud VM.

Why it matters:

- this is the cleanest predictable low-cost VPS path
- current Hetzner entry pricing remains low enough for a hobby or early research service, with `CAX11` listed at `EUR 4.49/month` in Germany and Finland as of the 2026 price update
- the operating model stays simple and fully self-managed

Key risks:

- it is not free
- once paid, cost discipline depends on keeping the topology intentionally small

### Option C: Cloudflare edge plus AWS EC2 instance

Keep the static web path unchanged, proxy `api.btcnetwork.info` through Cloudflare, and run the Rust API plus PostgreSQL on one EC2 instance.

Why it matters:

- EC2 satisfies the server-control requirement
- AWS currently offers up to `$200` in credits for new accounts during a six-month free account plan

Key risks:

- the current free path is credit-based and temporary, not an always-free VM stance
- the long-term cost floor is not the cheapest option evaluated here
- AWS adds more operational and billing surface than this repository needs in its first hosted API phase

### Option D: Managed API or managed PostgreSQL

Use a managed runtime, managed PostgreSQL, or a platform that removes direct VM control.

This is included only so it can be rejected explicitly.

## Proposed Decision

If accepted, this BNDD should set a two-tier provider strategy.

### Bootstrap target

Use `OCI Always Free` for the first public API deployment only when all of these are true:

- an account can actually be created and verified without unusual friction
- a suitable home region has available Always Free ARM capacity
- a small risk of idle reclamation is acceptable for the first public phase

Use one ARM VM for:

- reverse proxy
- Rust API
- PostgreSQL
- local backups and restore verification jobs

### Default paid fallback and steady-state low-cost path

Use `Hetzner Cloud` as the default paid path when:

- OCI capacity is unavailable
- OCI account setup blocks progress
- paid reliability and predictability matter more than preserving a `$0` origin bill
- the project wants to treat the API as a continuously available service rather than an experimental bootstrap

The intended first Hetzner target is a small ARM or shared-vCPU plan, not a larger general-purpose fleet.

### AWS stance

`AWS EC2` remains an allowed option, but not the default recommendation for this repository's first public API phase.

Use EC2 only when one of these becomes true:

- the project needs AWS-specific adjacency
- a later feature needs AWS networking or identity integration
- the team explicitly chooses AWS despite the higher cost floor

## Target Topology

```text
Browser
  -> btcnetwork.info
  -> Cloudflare Pages

Browser
  -> api.btcnetwork.info
  -> Cloudflare proxy / TLS / WAF / basic rate limiting / DDoS rules
  -> reverse proxy on VM
  -> Rust API on loopback
  -> PostgreSQL on loopback or local Unix socket
```

Initial origin layout:

- one VM
- one reverse proxy
- one Rust API systemd service
- one PostgreSQL instance
- one backup schedule

Explicit non-goals for the first slice:

- no Kubernetes
- no public PostgreSQL
- no separate database VM by default
- no managed PostgreSQL
- no direct client access to the origin IP

## Public Edge and DNS

When the API phase starts, Cloudflare should become the authoritative DNS provider for `btcnetwork.info`.

If accepted, this explicitly refines the future-phase DNS stance from [BNDD-0003](../BNDD-0003/BNDD-0003.md), which deferred authoritative DNS migration until a later operational reason existed.

Reason:

- Cloudflare documents that updating nameservers is required to activate a domain on Cloudflare and use most Cloudflare application services

Practical rule:

- keep domain registration at Hostinger unless there is a separate reason to move the registrar
- change nameservers from Hostinger DNS to Cloudflare DNS when the API goes live
- keep the frontend and API on the same Cloudflare zone so the web app and API share the same edge controls and certificate posture

The API hostname should be:

- `api.btcnetwork.info`

Origin exposure rules:

- Cloudflare-proxied DNS only
- no public PostgreSQL port
- firewall allows `80/443` only for Cloudflare ranges if operationally manageable
- firewall allows `22` only from known admin source IPs or a dedicated admin tunnel

## Rate Limiting and DDoS Posture

### Edge stance

Start with Cloudflare at the edge.

Cloudflare documents that:

- DDoS protection managed rulesets are enabled by default for zones onboarded to Cloudflare
- free-plan rate limiting is materially constrained compared with higher plans

That means the first posture should be intentionally narrow:

- one Cloudflare free-plan rate limiting rule for `/api/*`
- count by source IP
- use a free-plan-compatible counting period and threshold
- start conservative and tune upward from observed traffic

Current free-plan constraint that matters here:

- Cloudflare documents the free plan as materially narrower than higher plans, including one rule and a short counting period, so origin-side enforcement is still required

Because free-plan rate limiting is limited, the origin must still defend itself.

### Origin stance

Use the reverse proxy as a second control layer.

The reverse proxy should:

- trust client IP from Cloudflare headers only after Cloudflare networks are configured as trusted proxies
- apply a second per-IP request rate and burst cap
- return `429` for local-origin enforcement
- log the effective client IP and rate-limit actions

This dual-layer posture is deliberate:

- Cloudflare blocks a large share of low-effort abuse before origin
- origin still has protection if edge limits are too coarse or temporarily exceeded

### DDoS scope

This phase is about practical HTTP API protection, not perfect global attack resistance.

Acceptable initial posture:

- Cloudflare proxy for public HTTP access
- origin firewalling
- no direct database exposure
- no unsupported raw TCP service on the public hostname

Unacceptable posture:

- publishing the origin IP as a first-class endpoint
- exposing PostgreSQL to the internet
- relying only on origin-side rate limiting while leaving the edge unconfigured

## Database Placement and Backup Stance

The first database placement should be:

- PostgreSQL on the same VM as the API
- local disk or attached block volume
- database bound to local interfaces only

Why:

- compute plus block storage is the dominant fixed cost
- early traffic does not justify a second VM
- one-node operation is simpler to patch, back up, and restore

Backup stance:

- nightly logical backup
- periodic restore verification
- off-host encrypted backup copy
- periodic volume snapshot or provider-native volume backup

Provider-specific bias:

- on OCI bootstrap, provider-native object or volume backup features are acceptable because they fit the zero-dollar or near-zero-dollar goal
- on Hetzner, the cheapest storage add-on is acceptable for off-host copies if the API is already on paid infrastructure

Split PostgreSQL onto its own VM only when at least one of these becomes true:

- memory pressure from PostgreSQL and the API begins to conflict
- backup or restore windows become operationally unacceptable
- database disk growth outpaces the single-VM shape
- API deployment cadence starts to create unacceptable database coupling risk

## Rejected Approaches

### Reject: database public on the internet

This is not acceptable for the first deployment.

PostgreSQL should be private by default.

### Reject: origin-first protection with no edge proxy

This is too weak for a public anonymous API when the goal is low cost.

Without edge protection, abusive traffic lands directly on the smallest and cheapest part of the stack.

### Reject: starting with multiple VMs for symmetry

This optimizes for topology neatness, not for actual early-stage economics.

The simpler one-VM topology is easier to operate and cheaper to keep online.

### Reject: AWS as the default cheapest path

This BNDD does not reject AWS entirely.

It rejects AWS as the default first choice for this repository's low-cost public API phase because the free path is temporary credits and the steady-state paid floor is not the cheapest evaluated option.

## Revisit Conditions

Revisit this BNDD when one of these becomes true:

- API traffic is large enough that Cloudflare free-plan rate limiting is obviously insufficient
- the project needs more granular policy than IP-based anonymous limiting
- the single-VM host regularly runs near CPU, memory, or IOPS limits
- PostgreSQL backup retention no longer fits the chosen low-cost storage strategy
- OCI Always Free availability or idle-reclamation behavior makes the bootstrap path impractical
- legal, geographic, or latency constraints make the current preferred regions unsuitable
- the team decides that AWS ecosystem alignment matters more than the lower-cost VPS path
