## Purpose

This document chooses the first concrete deployment strategy for the public Rust API and its PostgreSQL database.

The repository already has a production path for the static frontend:

- `apps/web` on `Cloudflare Pages`
- browser-safe analytics reads through `apps/api`
- PostgreSQL as the default analytics storage backend

What was still open was the backend hosting direction:

- which cloud should host the first public API
- which IaC tool should define that infrastructure
- how API deployment and PostgreSQL should fit into CI/CD
- how the crawler should run when it is not a continuous always-on workload
- whether API and PostgreSQL should split before public launch
- how to preserve a future path to stronger edge caching and scale

This BNDD now settles those questions around an AWS-first direction.

## Decision Summary

The repository will use this strategy for the first hosted API phase:

- keep the static frontend on `Cloudflare Pages`
- use `AWS` as the primary cloud provider for the first public API and database deployment
- move authoritative DNS for `btcnetwork.info` to `Cloudflare` when the API goes live, while keeping domain registration at Hostinger unless a registrar move later reduces real operational friction
- keep `api.btcnetwork.info` behind a Cloudflare-proxied edge
- provision AWS infrastructure with `Terraform` CLI and keep the HCL compatible with `OpenTofu` where practical
- use one small public `EC2` instance for the reverse proxy, Rust API, and crawler trigger path
- use one separate private `EC2` instance for self-managed PostgreSQL from day zero
- keep PostgreSQL non-public and reachable only from the API/crawler security group over the private network
- treat the crawler as a sporadic batch workload on the API/crawler host, not as an always-on daemon by default
- use `systemd`-managed services on the VMs instead of container orchestration for this slice
- use AWS Systems Manager as the default operator path for host access and patching
- use Terraform plans in CI so infrastructure changes always produce a reviewable diff before apply
- allow Cloudflare free-plan edge caching from day one for safe anonymous read endpoints
- treat Cloudflare cache and optional edge compute as a later scale lever, not as a day-one requirement

## Why This Decision Exists

The user goals for this phase are specific:

- AWS credits make AWS a pragmatic choice now, even if it is not the absolute cheapest provider on paper
- Terraform is worth adopting both because it is widely used and because it helps make network primitives such as VPCs, subnets, route tables, and internet egress explicit
- the project still wants to preserve an eventual path away from AWS if operating constraints change
- infrastructure diffs must be reviewable before deployment
- CI/CD for both the API and PostgreSQL matters from the first hosted phase
- the first deployment should stay small enough to understand and operate alone
- the database should not inherit API deploy, crawler burst, or public-origin risk when the split is affordable from the start

Those goals point to one practical shape:

- Cloudflare at the public edge
- AWS for compute, networking, storage, and VM operations
- Terraform for declarative infrastructure
- one public API/crawler VM and one private PostgreSQL VM with explicit services and backups

This does not claim that two VMs are the ideal forever topology.

It claims that separating the database on day zero is the right first topology because the operational boundary is important and still easy to understand.

## Constraints

This decision must respect these constraints:

- the frontend deployment path already exists and should remain low-friction
- the API needs public internet reachability
- PostgreSQL should not be public
- the project wants to learn and own VM networking rather than hide behind a fully managed runtime
- the project wants IaC that can run in GitHub Actions and show a diff before apply
- the project wants to avoid deep platform lock-in where practical
- the crawler does not need to run continuously and should support scheduled or manual triggers
- the project should not introduce Kubernetes, ECS, a load-balanced API fleet, or a managed database in the first slice
- the project should not adopt managed PostgreSQL by default in the first slice

Two consequences follow:

1. The first deployment should optimize for simplicity, observability, and database isolation, not for theoretical horizontal scale.
2. Provider portability should come from generic VM patterns and clean Terraform boundaries, not from pretending every provider exposes identical primitives.

## Dominant Cost Drivers

For this phase, the dominant cost drivers are:

1. always-on `EC2` compute for two small instances
2. always-on `EBS` storage for PostgreSQL
3. backup retention and off-host copies

Request volume is not the first cost driver because:

- the frontend stays static at the edge
- the expected API workload is mostly read-heavy and cache-friendly
- Cloudflare can absorb part of the anonymous public traffic before it reaches origin

The first optimization should therefore be:

- keep the origin and database footprints small
- avoid introducing AWS managed services whose fixed cost dominates current traffic
- preserve a future cache layer so origin scale is not the first answer to a read-heavy workload

## Options Evaluated

### Option A: AWS single VM plus Cloudflare edge

Keep the current static web path, put `api.btcnetwork.info` behind Cloudflare, and run the Rust API plus PostgreSQL on one AWS EC2 instance in a small VPC.

Why it matters:

- matches the current AWS credit situation
- teaches the networking primitives the project wants to understand
- keeps the first backend small and explicit
- works cleanly with Terraform and GitHub Actions

Key risks:

- long-term AWS steady-state cost can be higher than a low-cost VPS provider after the credits are gone
- API and PostgreSQL share the same failure domain
- the single VM shape has limited isolation for storage, memory, and deploy risk

### Option B: AWS split API VM plus separate PostgreSQL node

Stay on AWS, but split the API and PostgreSQL into separate machines from day one.

Why it matters:

- cleaner isolation boundary
- safer restarts for the API tier
- clearer path toward independent scaling
- keeps crawler CPU, memory, and socket bursts away from PostgreSQL page cache and writeback behavior
- makes backup and restore operations a database-host responsibility instead of a side effect of the API origin

Key risks:

- more fixed cost immediately
- more networking and backup complexity before real traffic justifies it
- more operational surface for a one-operator system

### Option C: AWS managed services first

Use AWS compute plus managed services such as `RDS`, `ECS`, or `ALB` in the first hosted phase.

Why it matters:

- reduces some host-level maintenance work
- fits AWS-native service integration well

Key risks:

- higher fixed cost and more AWS-specific gravity
- weak fit for the current learning goal around VM networking and system operations
- more moving parts than the project needs today

### Option D: Different origin cloud provider

Keep Cloudflare at the edge but run the origin on another provider.

Why it matters:

- could lower long-term origin cost after AWS credits are exhausted

Key risks:

- contradicts the current explicit AWS preference
- does not help the current AWS learning goal
- can be revisited later without changing the first deployment shape

## Accepted Decision

The first public API deployment will use:

- `Cloudflare Pages` for the web frontend
- `Cloudflare` as the public API edge and future cache layer
- `AWS EC2` for the origin compute
- `AWS EBS` for PostgreSQL storage
- `Terraform` CLI as the default IaC runner
- HCL written to remain `OpenTofu` compatible where practical, so the configuration format is not tightly coupled to one runner

The first origin topology is intentionally small:

- one VPC
- one public subnet
- one private application subnet for the PostgreSQL host
- one public API/crawler EC2 instance
- one private PostgreSQL EC2 instance
- one reverse proxy on the API host
- one Rust API service on the API host
- one batch-style crawler service with explicit triggers on the API/crawler host
- one PostgreSQL service on the private database host
- one backup path

Explicit non-goals for the first slice:

- no Kubernetes
- no ECS or Fargate
- no `RDS`
- no multi-AZ database
- no `ALB`
- no API fleet or load balancer by default
- no direct public PostgreSQL access

## IaC Direction

Prefer `Terraform` as the canonical workflow and `Terraform`-style HCL as the canonical configuration format.

Reason:

- Terraform remains the most common shared IaC vocabulary across providers
- the project explicitly wants to learn it
- the plan workflow gives a first-class infrastructure diff before apply
- the same HCL can stay close to `OpenTofu`, which preserves an exit path if runner preferences change later

Practical rules:

- use `Terraform` CLI in GitHub Actions and local operator workflows
- keep provider-specific details isolated in dedicated root modules or stacks
- keep application deployment logic out of Terraform unless a tiny bootstrap step truly belongs there
- use remote state, not local state, for shared environments

Recommended first remote-state posture on AWS:

- store state in an `S3` backend
- enable bucket versioning
- enable backend locking with the `S3` lockfile mechanism
- add a lifecycle rule so noncurrent state versions expire quickly instead of accumulating long history
- keep the rollback window intentionally small, such as `7` days or a low retained noncurrent-version count such as `2`
- keep the state bucket and app infrastructure in separate Terraform roots so bootstrap remains simple

State-retention rule:

- do not treat Terraform state history as a long-term audit archive in this repository
- keep enough retained versions to recover from recent operator mistakes, but no more
- if stronger audit requirements appear later, solve that deliberately instead of keeping accidental state history forever

Portability rule:

- portability comes from staying on generic primitives such as VM, disk, DNS, firewall, and systemd service boundaries
- portability does not require pretending the AWS provider module will drop unchanged onto another cloud

## Operator Command Surface

Repository-level `make` targets are the default operator entrypoint for hosted
infrastructure tasks.

Reason:

- operators can run `make help` and discover the supported commands in one place
- CI and local commands can share the same target names where practical
- repeated Terraform, database, and machine-operation flags stay documented near the command
- apply and remote-machine actions can add guardrails without hiding what they run

Command groups should stay thin:

- `infra-tf-*` wraps Terraform format, init, validate, plan, and apply workflows
- `infra-linux-check` validates portable Linux host artifacts where local tools are available
- `infra-aws-*` wraps AWS Systems Manager access and common host service checks
- local development database targets such as `infra-postgres-up` remain separate from hosted PostgreSQL operations
- role docs under `infra/deployments/api/` and `infra/deployments/postgres/` define API and database deployment ownership separately

Guardrail rules:

- Terraform apply targets require an explicit confirmation variable such as `CONFIRM_APPLY=1`
- remote host targets require an explicit role instance id such as `API_SSM_INSTANCE_ID` or `POSTGRES_SSM_INSTANCE_ID`
- hosted database operations should go through Systems Manager or a documented deploy workflow, not broad public SSH
- Make targets should remain small wrappers; complex reusable logic belongs in scripts under `scripts/` or `infra/linux/scripts/`

The command surface is an operability contract, not a second infrastructure
definition. Terraform remains the source of truth for cloud resources, and
`infra/linux` remains the source of truth for host runtime artifacts.

## Target Topology

```text
Browser
  -> btcnetwork.info
  -> Cloudflare Pages

Browser
  -> api.btcnetwork.info
  -> Cloudflare proxy / TLS / WAF / rate limiting / future cache
  -> AWS public API EC2 reverse proxy
  -> Rust API on API host loopback
  -> PostgreSQL over private subnet
  -> AWS private PostgreSQL EC2
```

This shape keeps the first deployment explicit and cheap in moving parts.

Public-surface rule:

- the API host is public only for the reverse proxy entrypoint
- the Rust API process is not public and must listen only on API-host loopback
- the PostgreSQL host has no supported public client path
- PostgreSQL listens on a private interface or local socket on the database host
- the database security group allows `5432` only from the API/crawler security group
- the database host firewall must not expose `5432` beyond the private application network

## AWS Network and VM Baseline

The first AWS network should stay minimal:

- one VPC
- one public subnet
- one private application subnet for PostgreSQL
- one internet gateway
- one public route table with default internet egress for the API subnet
- private route behavior that lets the PostgreSQL host use AWS Systems Manager without public SSH
- one API/crawler EC2 instance
- one PostgreSQL EC2 instance with an attached EBS volume sized for PostgreSQL growth and backups

The API/crawler instance should have:

- an instance profile for AWS Systems Manager and deploy operations
- a security group that exposes only required public reverse-proxy ports
- a stable public entrypoint, preferably an Elastic IP, so DNS and operator expectations stay simple

The PostgreSQL instance should have:

- an instance profile for AWS Systems Manager and backup-related permissions
- no direct public client endpoint
- a security group that allows PostgreSQL only from the API/crawler security group
- EBS sizing and snapshot policy appropriate for database growth

Private management stance:

- prefer VPC interface endpoints for `ssm`, `ssmmessages`, and `ec2messages` so the database host can stay private
- add `s3` access through an endpoint or another documented path if backups upload directly to S3
- use NAT only if the cost and operational tradeoff is explicit

Public ingress stance:

- `443` is the steady-state public ingress port
- `80` is allowed only if needed for redirects or certificate bootstrap, and should not remain the primary serving path
- `22` should not be the default public admin path
- `5432` must never be public in this topology

Admin path stance:

- prefer `AWS Systems Manager Session Manager` as the default shell access path
- allow public SSH only if a later operational need is explicit and tightly scoped

Origin exposure rule:

- do not treat the API EC2 public IP as a supported public endpoint
- do not assign or document a public PostgreSQL client endpoint
- if Cloudflare-origin restrictions are automated safely, prefer limiting origin ingress to Cloudflare-published ranges before production traffic
- treat the public interface as a thin reverse-proxy boundary only, not as direct exposure of all local services

## Host Operations and VM Health

Both VMs should be treated as managed servers, not as opaque pet machines.

Use these defaults:

- bootstrap both hosts with `cloud-init` or similarly narrow first-boot scripts
- run the reverse proxy and API as explicit `systemd` units on the API host
- run PostgreSQL and backup timers as explicit `systemd` units on the database host
- run crawler oneshot/timer units on the API/crawler host by default
- use `Session Manager` for interactive access
- use `Patch Manager` for recurring OS patch policy
- use `CloudWatch` alarms for instance status, API health, database disk pressure, and backup health
- enable automatic EC2 recovery where supported for system-status failures

This is the intended “healthy maintenance” posture:

- infrastructure described by Terraform
- host access through AWS Systems Manager
- service supervision through systemd
- patch cadence through Patch Manager
- recovery signals through CloudWatch

That is enough operational structure for two small VMs without introducing a larger platform.

## Replicable Linux Host Baseline

The infrastructure plan should be explicit enough that the hosted stack can be reproduced on any ordinary Linux server, not only on AWS.

AWS remains the first provider, but the host-level runtime should stay portable across Linux environments.

That means the repository should eventually describe or materialize these concrete pieces:

- base package set
- system users and ownership model
- directory layout for releases, config, logs, and backups
- reverse proxy config
- API environment file contract
- `systemd` unit files and timers
- PostgreSQL config files
- firewall rules
- backup scripts

Minimum Linux host artifact inventory:

- role-specific `cloud-init` or equivalent first-boot provisioning steps
- package list for the chosen distro, such as reverse proxy, PostgreSQL, backup tooling, and operator utilities
- dedicated service users for API/crawler workloads and database/backup workloads
- release directory convention, for example `/opt/btc-network/api/releases/<version>`
- active symlink or equivalent runtime path, for example `/opt/btc-network/api/current`
- runtime environment file location, for example `/etc/btc-network/api.env`
- reverse proxy site config
- API `systemd` service
- crawler `systemd` oneshot service
- crawler `systemd` timer if recurring host-local scheduling is used
- backup `systemd` service and timer
- PostgreSQL `postgresql.conf`
- PostgreSQL `pg_hba.conf`
- Linux VM tuning file for PostgreSQL memory and writeback behavior, for example `/etc/sysctl.d/90-btc-network-postgres-memory.conf`
- `systemd` resource-control overrides for API/crawler services on the API host
- host firewall rules for the public API host and private PostgreSQL host

PostgreSQL configuration stance:

- keep `postgresql.conf` and `pg_hba.conf` explicit and reviewable
- prefer a minimal, commented baseline over distro defaults that are never written down
- document the private bind address, socket usage, auth method, memory-related settings, WAL/retention choices relevant to backups, and logging posture
- treat Linux page cache as part of the PostgreSQL memory budget, not as “free” RAM left over after `shared_buffers`
- document reclaim and writeback posture explicitly, including swap stance, `vm.swappiness`, and dirty-page thresholds or byte limits relevant to EBS-backed write latency
- keep the database host free of API and crawler workloads by default so PostgreSQL page cache and writeback behavior are easier to reason about

Replication rule for infrastructure docs:

- every host-level behavior that matters to production should map to a concrete file, unit, rule, or command
- do not leave critical setup as dashboard memory or undocumented shell history
- if a piece cannot be reproduced on a plain Linux host with repository docs and managed config artifacts, treat the design as incomplete

AWS-specific services may still exist around that baseline:

- VPC, security group, IAM role, SSM, CloudWatch, and EBS remain AWS-specific
- the Linux service layout, PostgreSQL config, reverse proxy config, and deployment unit model should remain portable

## API and PostgreSQL Deployment

The deployment shape should stay explicit and role-separated:

- API host: reverse proxy on the public interface
- API host: Rust API bound only to loopback
- API/crawler host: crawler oneshot/timer units connect to PostgreSQL over the private network
- PostgreSQL host: PostgreSQL bound only to a private interface or local Unix socket
- PostgreSQL host: backup service and timer own logical backup execution
- migrations run as a controlled deploy step, not by ad hoc shell use

This means the public API host and database host have different deployment responsibilities:

- API deployment ships the Rust API artifact, API environment file references, reverse proxy updates, and API `systemd` changes
- PostgreSQL deployment ships PostgreSQL config, host tuning, backup scripts, backup timers, and migration execution policy
- crawler deployment ships the crawler artifact and trigger/timer units on the API/crawler host

The important distinction is:

- public API reachability does not imply public database reachability
- database reachability is controlled by private bind address, security group source rules, host firewall policy, and PostgreSQL `pg_hba.conf` together

The API and database should split on day zero because:

- the database does not inherit API deploy risk or reverse-proxy exposure
- PostgreSQL memory, page cache, checkpoints, and backups are easier to reason about on a dedicated host
- crawler bursts do not evict PostgreSQL cache or contend with database writeback on the same kernel
- backup and restore workflows can be practiced against a clear database host boundary
- the extra topology is still small enough for one operator: two VMs, not a platform

The main tradeoff is clear:

- isolation, backup clarity, and future scale posture improve
- fixed cost and network/configuration complexity increase

That tradeoff is acceptable because AWS credits reduce the immediate cost concern and the database boundary is a real operational boundary, not speculative architecture.

Database-host memory-budget rule:

- size the database host for PostgreSQL shared memory, expected per-query or per-connection memory, Linux page cache, backup overhead, and OS headroom together
- do not treat PostgreSQL RSS alone as the real database footprint on Linux; page-cache growth and reclaim behavior matter directly to latency stability
- keep eviction posture conservative enough that the kernel does not churn hot PostgreSQL pages under ordinary query and backup load

## Deployment Availability and Rollouts

In the first split-VM topology, API deployment and PostgreSQL operations are separate workflows.

The simplest API deploy path can cause brief API downtime:

- stop or restart the existing API systemd unit
- replace the binary or release artifact
- start the updated service on the same port

That is acceptable in the earliest phase when traffic is still modest and operator simplicity matters more than continuous availability.

The preferred next step before multi-node scale is same-host blue/green deployment:

- keep the reverse proxy on the public interface
- run the current API instance on one local port
- start the new API version on a second local port
- health-check the new instance locally
- switch the reverse proxy upstream to the new port
- drain or stop the old instance after the switch

Why this is the preferred intermediate step:

- much lower complexity than multi-node rolling deploys
- near-zero-downtime application cutover on the API host
- no need for a load balancer yet
- keeps the repository on the same simple API-host plus database-host model

Important limits:

- same-host blue/green does not remove the API host failure domain
- it does not solve capacity limits
- it does not make incompatible database migrations safe by itself

Database rollout rule:

- prefer backward-compatible expand/contract migrations when a schema change spans old and new API versions
- do not rely on blue/green app switching alone to make a breaking schema change safe

Escalate beyond same-host blue/green when one of these becomes true:

- deploy frequency or uptime goals no longer tolerate single-host risk
- uncached traffic requires more than one API instance for capacity
- the API needs rolling deploys across failure domains rather than only fast local cutover

Backup stance:

- use PostgreSQL logical backups for schema-level recovery
- use EBS snapshots for volume-level recovery
- automate snapshot retention rather than relying on manual operator discipline
- keep at least one off-host copy path beyond the running instance
- verify restore steps regularly instead of assuming backups are valid

## Crawler Scheduling and Triggers

The crawler should not be treated as a permanently running service by default.

The accepted first-hosted posture is:

- run the crawler on the API/crawler EC2 instance as a `systemd` oneshot service
- trigger it either from a local `systemd` timer for recurring runs or from `AWS Systems Manager Run Command` for manual or operator-driven runs
- keep `EventBridge -> SSM Run Command` as an allowed later path if the schedule should live in AWS control-plane configuration instead of on the VM

Default recommendation:

- use a local `systemd` timer when the cadence is simple and host-local scheduling is enough
- use `SSM Run Command` for ad hoc runs, operational retries, or admin-triggered collection

This keeps the first design small:

- no separate scheduler service
- no always-on crawler supervisor
- no extra queue or workflow engine

Important deployment rule:

- never allow overlapping crawler runs against the same PostgreSQL database

Reason:

- this is an intentional design choice, not an accidental bug
- keeping one active crawler writer lets concurrency and parallelism stay inside one process where the crawler already controls them explicitly
- this avoids unnecessary cross-process locking, coordination, and contention in the application layer
- the crawler no longer depends on durable startup recovery, so each run should be treated as a fresh batch from seeds unless a later design changes that

Enforcement direction:

- the trigger path should take a host-level or database-backed lock before starting the crawler
- if a run is already active, the second trigger should fail fast and log a clear reason instead of starting another crawler instance

Operational tradeoff on the API/crawler host:

- the scheduler model stays simple
- crawler CPU and memory pressure can still compete with the API process
- PostgreSQL is protected from crawler CPU and page-cache contention by the separate database host

Because of that:

- keep crawler cadence conservative at first
- prefer running heavier crawls outside expected peak API usage windows
- revisit a separate crawler/worker host before adding more scheduling complexity if batch pressure becomes the dominant source of contention

## CI/CD and Infrastructure Diff Workflow

CI/CD needs to cover both code and infrastructure.

The default workflow should be:

1. PRs run build, test, and security checks for the repository.
2. Infra PRs also run `terraform fmt -check`, `terraform init -backend=false`, `terraform validate`, and `terraform plan`.
3. The plan output is attached to the PR or workflow artifacts so the infrastructure diff is reviewable before merge.
4. Protected-branch deploy workflows apply infrastructure only after review.
5. API deploy workflows build the Rust binary, publish a versioned artifact, and update the VM through a controlled channel such as Systems Manager Run Command or a tightly scoped SSH workflow.

Important Terraform review rule:

- plan files can contain sensitive values in cleartext, so treat them as sensitive CI artifacts

Application deploy rule:

- infrastructure provisioning and application rollout should be separate workflow concerns even if the same pipeline triggers both

Recommended separation:

- `infra plan/apply` for AWS and Cloudflare resources
- `api deploy` for shipping the Rust API binary, API environment file references, reverse proxy changes, and API systemd restart steps
- `postgres deploy` for PostgreSQL config, host tuning, backup scripts/timers, snapshot policy, and restore validation
- `crawler deploy` for crawler binary and timer/trigger units on the API/crawler host

## Scaling and Split-Topology Revisit Triggers

Two small VMs are a good first fit when all of these are true:

- traffic is still modest
- the API is mostly read-heavy
- the cache hit opportunity is high
- one operator needs a simple mental model with clear API/database boundaries

The dominant scale risk is not raw CPU first.

The dominant early risks are still shared-resource and boundary risks:

- the API and crawler compete for RAM and CPU on the API/crawler host
- PostgreSQL durability and query spikes compete for disk IOPS and throughput
- network rules or migration mistakes can still affect both tiers
- the database host can become the first bottleneck even if the API host is healthy

Keep the split topology until measured pressure says otherwise.

Revisit the topology when one or more of these becomes true:

- PostgreSQL memory, page cache, or disk pressure requires a larger database host or managed database
- backup/restore windows become too slow for the recovery objective
- API/crawler contention becomes visible on the API host
- API deploy cadence requires more than same-host blue/green
- one database host can no longer meet the recovery objective after failure
- uncached traffic volume justifies multiple API instances

Preferred scale order:

1. improve cacheability and edge cache hit rate
2. vertically scale the constrained host or EBS volume
3. move crawler/batch work to a separate worker host if it competes with API latency
4. add a second API node and a load balancer when uncached request volume actually requires it
5. revisit managed PostgreSQL only when self-hosted database operations cost more than they save

## Future Edge Cache Direction

This project should not start with edge compute as a hard dependency.

The first cache move should be simpler:

- start with Cloudflare free-plan caching for public anonymous `GET` endpoints that are explicitly safe to cache
- make API `GET` responses cache-friendly
- use explicit `Cache-Control` and validator headers where the API contract allows it
- add `Cache-Tag` support for read endpoints that may need targeted invalidation later
- use Cloudflare Cache Rules or Cache Response Rules to cache safe public reads
- purge by URL or cache-tag from the origin when a write or admin action invalidates cached data

Why this is the right first future path:

- the data is highly cacheable
- the free Cloudflare cache feature set is already sufficient for an early public read-heavy API
- CDN caching is simpler than moving logic to edge code
- Cloudflare already fronts the public hostname

Free-plan constraints that still matter:

- keep caching limited to public anonymous responses
- keep cache rules narrow and explicit
- respect free-plan purge-rate limits when designing invalidation behavior
- do not depend on advanced cache analytics or enterprise-only cache-key features in the first slice

Edge-compute stance:

- `Cloudflare Workers` free plan exists and is viable for small future edge logic
- do not make Workers the primary API runtime until there is a concrete need for edge-side request shaping or composition

For now, the design goal is narrower:

- origin remains the source of truth
- edge cache can absorb hot public reads later
- invalidation should be explicit and origin-driven

## Rejected Approaches

### Reject: treating AWS managed services as the default first step

This is not the right first move for this repository.

Managed services can be revisited later, but they raise fixed cost and reduce the hands-on VM and network learning value the user wants now.

### Reject: co-locating API and PostgreSQL on day zero

This is cheaper and simpler on paper, but it couples public-origin operations,
API deploys, crawler bursts, and PostgreSQL memory/writeback behavior too early.

Because AWS credits reduce immediate cost pressure, the first deployment should
pay the small complexity cost for a clearer database boundary.

### Reject: local Terraform state for shared environments

This is too brittle for CI/CD and reviewable operations.

Remote state is required.

### Reject: making portability the primary optimization

The current decision is intentionally AWS-first.

Portability still matters, but it should be preserved through generic boundaries and Terraform structure, not by weakening the initial design into the lowest common denominator.

### Reject: adopting edge compute before plain caching

The data shape does not justify that complexity yet.

Caching should come before edge business logic.

## Revisit Conditions

Revisit this BNDD when one of these becomes true:

- AWS credits are near exhaustion and steady-state cost becomes the main driver
- either host regularly runs near memory, CPU, or storage limits
- backup or restore windows become operationally unacceptable
- the project needs stronger availability than one API host or one database host can provide
- a managed PostgreSQL service becomes cheaper in operator time than self-hosting
- the API contract changes in a way that makes edge compute materially more valuable than plain caching
- another provider becomes strategically important enough to justify a new origin migration BNDD
