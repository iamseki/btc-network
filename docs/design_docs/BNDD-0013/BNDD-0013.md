## Purpose

The public web app needs a dedicated `Status` surface for specific Bitcoin network endpoints that the project deliberately monitors.

This status page is not a general uptime monitor for arbitrary hosts. It should answer a narrower question: are selected Bitcoin DNS seeders and selected community-known public nodes currently reachable enough for this project to observe them?

## Decision Summary

The repository will add a curated node status feature:

- add a `Status` section to the `Network Analytics` home page
- expose a direct `/status` browser path for the full status page
- expose read data from `GET /api/nodes/status`
- keep the API response as a simple array of target status rows
- run status checks through the crawler app as a separate `status-check` subcommand
- load checked targets from a local config file rather than sampling arbitrary discovered nodes
- persist compact append-only status history in PostgreSQL instead of only keeping the latest 5-minute view
- show stale status in the frontend when the newest check for a target is older than `5 minutes`
- change a target to `failed` only after the status runner exhausts `5` resolution/handshake attempts for that check
- keep demo mode available in the web client with deterministic status rows

This decision is accepted and should guide implementation.

## Why This Decision Exists

The existing crawler analytics surface describes the broader observed network. It does not provide a small, operator-friendly view of known bootstrap and public endpoints.

The dominant cost driver for this feature is outbound network checking, not database reads. Keeping checks in a scheduled crawler subcommand avoids doing handshakes during API requests and keeps the web/API path cheap and predictable.

The status page should be simple enough to trust:

- specific targets are named and documented
- each target has a recent check timestamp
- each target can show recent historical health, not only a single latest row
- short network flakes do not immediately become public failures
- stale data is visibly different from current failure data

## Decision Details

### Curated Targets Only

Status checks use a checked-in or operator-provided target config file. Every target is a specific endpoint, and that endpoint is the stable identity for API grouping and display.

The status runner must not pick arbitrary node addresses from the crawler frontier, latest observations, DNS seed responses, or third-party APIs unless those addresses are explicitly added to the config.

For every configured target, the status runner should:

- resolve the configured host through normal DNS resolution
- attempt a Bitcoin version handshake with the configured endpoint
- use the existing shared session/protocol paths
- keep status logic in crawler/status workflow code rather than API handlers or React components

### Retry And Failure Policy

Each status check cycle records one final result per configured target.

Per cycle:

- attempt DNS resolution and handshakes with bounded timeout and backoff
- allow up to `5` resolution/handshake attempts before the target is marked `failed`
- reset status to `healthy` on the next successful check
- keep the latest failed message concise and public-safe

The retry budget is the first-slice protection against normal network flakiness. Do not add extra incident or escalation state until operational evidence shows the page needs it.

### Config File

Use a small TOML file.

Example:

```toml
[[targets]]
endpoint = "seed.bitcoin.sipa.be:8333"
label = "Sipa DNS Seed"
description = "Long-running Bitcoin Core mainnet DNS seed."

[[targets]]
endpoint = "seed.bitnodes.io:8333"
label = "Bitnodes Seed"
description = "Bitnodes community seed host used by btc-network for public sampling."
```

The repository should ship DNS seeders by default. Public node entries should be curated deliberately, with descriptions explaining why the project monitors each one. Do not use Bitnodes address lists or crawler-discovered addresses as default targets.

## API Contract

Add:

```text
GET /api/nodes/status
```

Return a simple array:

```json
[
  {
    "endpoint": "seed.bitcoin.sipa.be:8333",
    "label": "Sipa DNS Seed",
    "description": "Long-running Bitcoin Core mainnet DNS seed.",
    "status": "healthy",
    "checkedAt": "2026-05-02T12:00:00Z",
    "message": "Handshake succeeded.",
    "history": [
      { "status": "healthy", "checkedAt": "2026-05-02T12:00:00Z" },
      { "status": "healthy", "checkedAt": "2026-05-02T11:59:30Z" },
      { "status": "failed", "checkedAt": "2026-05-02T11:59:00Z" }
    ]
  }
]
```

Field notes:

- `status`: `healthy`, `failed`, or `unknown`
- `checkedAt`: newest persisted check result for the target
- `message`: short public-facing summary, not raw internal error spam
- `history`: recent status points for the same endpoint, ordered newest first; default API window is the last `24 hours`

The endpoint should not return a top-level summary, `generatedAt`, or stale threshold. The frontend can derive page summary, recent health, and stale state from each row.

## Storage Shape

Use one compact append-only status history table. Do not duplicate node observation fields that already belong in `node_observations`.

Suggested table:

```sql
CREATE TYPE NODE_STATUS AS ENUM ('healthy', 'failed', 'unknown');

CREATE TABLE node_status (
  node_status_id bigserial PRIMARY KEY,
  endpoint text NOT NULL,
  label text NOT NULL,
  description text NOT NULL,
  status NODE_STATUS NOT NULL,
  checked_at timestamptz NOT NULL,
  message text NOT NULL
);

CREATE INDEX node_status_endpoint_checked_at_idx
  ON node_status (endpoint, checked_at DESC);
```

This table intentionally stores status-page history only, not full node observations. Detailed protocol observations remain in `node_observations`.

History growth is linear in `targets * checks_per_day`. With roughly 10 targets and 30-second checks, the table receives about 28,800 rows per day. That is acceptable for the first slice, but the implementation should include an operator retention setting with a conservative default such as `30 days` so this table does not grow indefinitely. API reads should return only a bounded recent window, defaulting to `24 hours` per endpoint.

## Frontend Behavior

The web app should add:

- a `Status` entry under the network analytics group
- a `/status` direct browser path
- a compact home section that links to the full status page
- a full status page that polls `GET /api/nodes/status` every `30 seconds`
- deterministic demo data when `VITE_DEMO_MODE` is enabled
- recent history markers per target, derived from each row's `history`

Stale handling:

- if `Date.now() - checkedAt > 5 minutes`, show the target as stale/orange even if last stored status is healthy
- show clear copy such as `No fresh status check in the last 5 minutes`
- keep failed/red for fresh `failed` rows

Visual direction:

- adapt shadcn status blocks and table/list primitives to the existing black/gold instrument-panel style
- use compact service-style rows inspired by status monitoring pages such as AIWatch
- avoid large marketing copy; this is an operational status surface

## Tradeoffs

### Polling Versus Streaming

Polling every `30 seconds` is the chosen first implementation.

Reason:

- low target count
- API returns a small read-only JSON array
- browser implementation stays simple
- no need for server-sent events or websocket lifecycle management yet

If target count or update cadence grows enough that polling becomes material, revisit with SSE.

### Latest-Only State Versus History

The first slice stores compact status history rather than only latest status.

Reason:

- a single latest row only answers whether a target was reachable recently
- recent healthy history helps users distinguish a one-off failure from a consistently reachable target
- `node_observations` already stores detailed observation data
- the status history table stays intentionally small and display-focused
- retention keeps storage growth bounded without adding a heavier incident system

## Consequences

Positive consequences:

- users can inspect known Bitcoin bootstrap/public endpoints from `/status`
- public API stays cheap and read-only
- status checks reuse crawler/runtime networking boundaries
- stale status is explicit instead of silently showing old healthy data
- recent health can be displayed without scanning full crawler observation history

Tradeoffs:

- public node list requires curation and maintenance
- DNS seeder status is a proxy based on resolved peer handshakes, not direct seeder uptime
- compact status history needs retention policy and indexed latest-row queries
