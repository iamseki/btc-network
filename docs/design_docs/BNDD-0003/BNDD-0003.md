## Purpose

This document chooses the deployment direction for the public web frontend now, while keeping a pragmatic path open for a future Rust API and a self-managed PostgreSQL database.

The goal is not to find one perfect platform for every future concern. The goal is to choose the simplest deployment approach that:

- costs nothing for the initial low-traffic web phase
- keeps security defaults strong
- fits the current web-first architecture
- does not block a later API and PostgreSQL deployment
- avoids unnecessary vendor lock-in

`BNDD-0005` later makes crawler control and analytics the first concrete feature set that requires that Rust API. That refines the timing of the API need, but it does not change the deployment direction chosen in this document.

## Decision framing

The repository already has:

- a web-first SPA in `apps/web`
- GitHub Actions CI
- a future direction that includes a Rust API
- a settled storage direction toward self-managed PostgreSQL from `BNDD-0007`

Those facts matter because they make the deployment problem naturally staged:

1. deploy the static web application first
2. defer API and database infrastructure until they are real product needs
3. choose day-one tooling that still works when the project grows into a small self-hosted backend

`BNDD-0005` narrows that general future API into a concrete next-phase need for browser-backed crawler control and analytics reads. The hosting choice here still stands.

Trying to force the web, API, and PostgreSQL into one platform from day one would optimize for symmetry instead of the actual product phase.

That is not pragmatic for this repository.

## Constraints

The deployment choice must respect these constraints:

- day-one cost target is `$0`
- low traffic is expected initially
- security and cost are the main drivers
- the project already owns the `btcnetwork.info` domain at Hostinger
- the web app should stay a static SPA for the first phase
- the future API is in Rust and expected to use low resources
- PostgreSQL must be self-managed
- the project should prefer low-complexity and low-lock-in decisions

Two practical implications follow:

- zero cost is realistic for the static web phase
- zero cost is not a realistic long-term requirement once a self-managed API and PostgreSQL database exist

## Options evaluated

### Option A: Cloudflare Pages now, Hetzner Cloud later

Use Cloudflare Pages for the static SPA today, GitHub Actions for CI/CD, and keep the future API and PostgreSQL deployment on self-managed Hetzner Cloud infrastructure once the project accepts paid hosting.

This is the staged split-provider option.

### Option B: Vercel now, Hetzner Cloud later

Use Vercel Hobby for the web now and still plan to move the API and PostgreSQL deployment to a self-managed VM later.

This keeps the same future backend shape as Option A, but changes the frontend platform.

### Option C: Render static site now, future VM later

Use Render’s free static site offering now and choose a separate future host for API and PostgreSQL later.

This is viable, but it is weaker on the “clean free static hosting” fit than Option A.

### Option D: One provider for everything from day one

Choose one platform now for the web, future Rust API, and future self-managed PostgreSQL.

This is included only so it can be rejected explicitly.

## Evaluation criteria

The decision uses these criteria, in priority order:

1. `Cost at low traffic`
2. `Security`
3. `Operational simplicity`
4. `Fit for future Rust API + self-managed PostgreSQL`
5. `Portability / lock-in`

For scoring, use `5 = strongest fit` and `1 = weakest fit`.

## Decision matrix

| Option | Cost at low traffic | Security | Operational simplicity | Future API + self-managed PostgreSQL fit | Portability / lock-in | Notes |
| --- | --- | --- | --- | --- | --- | --- |
| `Cloudflare Pages + Hetzner later` | `5` | `5` | `5` | `5` | `4` | Best match for free static hosting now and simple VM path later |
| `Vercel + Hetzner later` | `4` | `4` | `5` | `5` | `3` | Good DX, but more product gravity toward platform-specific features and Hobby limits |
| `Render static + future VM` | `3` | `4` | `3` | `4` | `4` | Viable, but free offering is framed more as preview/hobby and shares usage limits |
| `Single-provider from day one` | `2` | `3` | `2` | `2` | `2` | Optimizes for symmetry, not for zero-cost start plus future self-managed stateful services |

### Why Option A wins

It wins because it fits the current phase without pretending the future API and PostgreSQL phase have the same constraints.

More concretely:

- Cloudflare Pages gives a real free path for static delivery with HTTPS, CDN, and Git integration already aligned with a low-traffic SPA
- GitHub Actions is already present, so CI/CD does not need a second workflow system
- Hetzner Cloud is a pragmatic future default for a low-resource Rust API and self-managed PostgreSQL deployment because it gives direct VM control and predictable low monthly cost
- Terraform/OpenTofu keeps the infrastructure description portable across Cloudflare and future VM providers

## Decision

The repository should adopt this staged deployment strategy:

### Phase 1 decision

- Deploy only `apps/web`
- Use `Cloudflare Pages` on the free tier
- Keep the app as a static SPA artifact
- Use `GitHub Actions` as the CI/CD trigger and delivery workflow
- Use `Terraform/OpenTofu` as the IaC direction for real web-edge resources only
- Keep domain registration at Hostinger for `btcnetwork.info`
- Use `btcnetwork.info` for the web app and reserve `api.btcnetwork.info` for the future API

### Phase 2 decision

- Add the Rust API only when it is a real product need
- Host the first API deployment on a self-managed VM
- Treat `Hetzner Cloud` as the default target for that first paid phase
- Run PostgreSQL as self-managed infrastructure, not as a day-one concern and not on a frontend-first platform

### Default topology for the first paid phase

One VM is acceptable at the start if it keeps the system simple:

- reverse proxy
- Rust API service
- PostgreSQL service
- backups and monitoring

Split the API and PostgreSQL across separate nodes only when reliability, storage growth, or operational isolation clearly justify it.

## Target architecture

### Phase 1

```text
Browser
  -> btcnetwork.info
  -> Cloudflare Pages
```

### Later phase

```text
Browser
  -> btcnetwork.info / api.btcnetwork.info
  -> Cloudflare DNS/CDN
  -> Rust API on VM
  -> PostgreSQL on the same first VM or an adjacent VM
```

This keeps the frontend simple now and preserves a straightforward future path.

## IaC direction

Prefer `Terraform/OpenTofu`.

Use Terraform-style configuration as the primary description format because it is the clearest shared reference for most engineers and providers, while keeping the implementation compatible with OpenTofu as the default open-source execution path.

Practical rules:

- keep CI/CD orchestration in GitHub Actions
- keep IaC focused on infrastructure state, not build logic
- separate provider concerns cleanly
- do not provision speculative future resources just to keep the repository “symmetrical”

The first IaC scope should be narrow:

- Cloudflare Pages project
- `btcnetwork.info` DNS records needed for the web app
- minimal environment-specific configuration

Future VM/network/firewall resources should live in separate modules or stacks when the API phase begins.

### Domain default

The domain is already registered at Hostinger.

The default should therefore be:

- keep domain registration at Hostinger
- point `btcnetwork.info` to Cloudflare Pages for the web frontend
- reserve `api.btcnetwork.info` for the future Rust API
- defer any full authoritative DNS migration to Cloudflare unless it later reduces real operational friction

This avoids unnecessary registrar or DNS migration work in the current phase.

## Security defaults

### Phase 1

- static frontend only
- HTTPS by default
- no long-running servers to patch
- no public database
- no public API surface yet
- minimal DNS change surface because the existing domain stays at Hostinger unless a later migration is justified

This is the lowest-risk deployment shape for the current product phase.

### Future API phase

- use SSH keys, not passwords
- expose only the minimum public ports
- put the API behind a reverse proxy and TLS
- keep PostgreSQL non-public by default
- restrict PostgreSQL access to the local host, private network, or explicitly allowed API host
- manage deployment credentials through GitHub Actions secrets or OIDC where supported

The important security principle is simple: do not introduce a public stateful service until the product actually needs one.

## Cost stance

The cost target should be explicit:

- `Phase 1`: `$0`
- `Phase 2`: low monthly infrastructure cost is acceptable

This document intentionally does not pretend that “free forever” is realistic once a Rust API and a self-managed PostgreSQL deployment exist.

That future phase will incur at least:

- VM cost
- storage cost
- backup cost
- operator time

The pragmatic move is to keep the initial web deployment free, then accept a small paid infrastructure baseline once the backend becomes real.

## Phased rollout

### Phase 1: web-only deployment

- build `apps/web`
- deploy to Cloudflare Pages
- keep CI/CD in GitHub Actions
- add only the minimum Pages/DNS IaC needed
- point `btcnetwork.info` at the Pages deployment without changing the domain registrar

### Phase 2: first backend deployment

- introduce the Rust API as a separate deployable service
- provision a small Hetzner Cloud VM
- deploy the API and PostgreSQL with simple service management
- add backups, firewall rules, and basic monitoring
- publish the API under `api.btcnetwork.info`

### Phase 3: scale only when justified

- split API and PostgreSQL if storage, reliability, or maintenance pressure justifies it
- add stronger observability and recovery automation
- revisit provider choice only if real operating constraints appear

## Rejected approaches

### Rejected: one platform for everything from day one

Reason:

- it pushes a future backend problem into the current web-only phase
- it usually increases lock-in pressure
- it does not improve the zero-cost day-one goal
- it is a poor fit for self-managed PostgreSQL

### Rejected: deploy the API and PostgreSQL immediately just to “prepare for the future”

Reason:

- it introduces cost before the product needs it
- it increases the attack surface
- it adds maintenance work without current product value

### Rejected: choose a frontend platform by backend convenience alone

Reason:

- the current deliverable is a static SPA
- the best static hosting decision should be made for that fact
- the future API and PostgreSQL phase should be solved with infrastructure that matches stateful workloads
