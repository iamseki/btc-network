## Purpose

The public API already exposes a generated OpenAPI document. That document is the right source of truth for exact endpoints, schemas, parameters, status codes, and error shapes.

Agents still need a smaller operational guide that answers how to use the API well: what to call first, what to cache, how to keep reads cheap, and when to stop guessing and inspect OpenAPI.

## Decision Summary

The API service will serve a hand-authored Markdown guide at:

```text
GET /agents.md
```

The guide is documentation only. It does not introduce new API behavior, auth, billing, protocol parsing, or crawler logic.

The web home will expose the guide near the current OpenAPI/API call to action as an integration-docs band. The homepage is the only rendered UI surface for this guide in the first slice; the API page keeps its existing Docs, Overview, and Access panels.

## Decision Details

`/agents.md` lives with the API docs module because the API service already owns `/api/openapi.json`, `/api/docs/config.json`, and `/docs`.

The route returns static Markdown with:

```text
Content-Type: text/markdown; charset=utf-8
```

The guide must stay lightweight and hand-authored rather than generated from OpenAPI. OpenAPI remains canonical for machine-readable contract details.

## Guide Content

The guide should tell agents to:

- start with `GET /api/v1/network/historical/runs?limit=1` to find the latest finished snapshot
- call `GET /api/v1/network/historical/runs/{run_id}` only when run detail is needed
- use last-run distribution endpoints for compact current analytics
- use `GET /api/nodes/status` for curated endpoint health
- start with low `limit` values such as `5`, `10`, or `20`
- avoid frequent broad reads of `GET /api/v1/network/last-run/nodes?limit=1000`
- treat current pagination as bounded `limit` parameters, not cursor pagination
- cache `/agents.md` and `/api/openapi.json` until deployment changes
- cache historical run details by `run_id`
- refresh latest-run analytics only after discovering a newer run
- treat `/api/nodes/status` as short-lived status data
- use `/api/openapi.json` for exact schemas, parameters, errors, and status codes

The guide should also tell agents not to:

- use frontend pages as API contracts
- scrape Scalar docs
- guess fields not present in OpenAPI
- call desktop-only single-peer workflows from browser/API integrations

## Frontend Behavior

The `Network Analytics` homepage should update the existing API area into a compact integration-docs band with two neighboring actions:

- OpenAPI reference/spec
- Agent guide page

The agent guide action should open a dedicated in-app `Agent Guide` view under the API center navigation, not a second preview card beside the OpenAPI CTA and not a new sidebar navigation item. That view should load the Markdown guide, show it in a code-style Markdown block, include a copy button, and include a raw `agents.md` link.

Normal builds should load the API-hosted `/agents.md` URL. In demo mode, the page should render a bundled Markdown import from `src` without making any network request. The web import should be a symlink to the API guide so there is one maintained Markdown source.

The maintained Markdown source is `apps/api/src/docs/agents.md`. The web symlink at `apps/web/src/lib/api/api-agents.md` exists only so Vite can bundle the same content for API-free demo mode.

Do not add a new API page tab for the guide in this slice.

## Tradeoffs

This design duplicates some guidance that could theoretically be inferred from OpenAPI descriptions. That duplication is acceptable because OpenAPI is optimized for exact contract lookup, while agents benefit from a short operational route map.

The guide is static for now. This keeps the implementation maintainable and avoids adding generator logic before the API surface is large enough to need it.
