## Purpose

This document replaces the repository’s default crawler persistence direction.

The project previously chose ClickHouse from benchmark evidence and implemented that path. That work remains useful, but it is no longer the default direction for new work.

The current goal is narrower and more pragmatic:

- reduce self-hosted infrastructure cost
- fit comfortably on a smaller VM
- optimize for operator familiarity and faster iteration
- lower cognitive load while the project is still experimental
- keep the shared crawler boundary stable while swapping the default adapter

This decision is now implemented in the repository.

## Decision Summary

The repository will adopt this direction:

- use plain `PostgreSQL` as the default database for crawler observations, checkpoints, and analytics reads
- keep `crates/btc-network` as the shared home for crawler domain types, workflows, `CrawlerRepository`, and `CrawlerAnalyticsReader`
- keep backend-specific SQL, migrations, and repository code outside the shared crate
- use explicit SQL with a small async connection pool
- use forward-only checked-in SQL migrations
- do not add a runtime backend switch or dual first-class backend support in this slice
- do not require any historical data migration because the project is still experimental

## Why This Decision Exists

`BNDD-0002` answered a benchmark question well, but that is not the only question that matters for the current project phase.

The current decision drivers, in order, are:

1. lower self-hosted cost and smaller VM footprint
2. operator familiarity and faster iteration from deep PostgreSQL experience
3. lower cognitive load and better maintainability
4. sufficient performance for the project’s current experimental stage

This is not a claim that ClickHouse is wrong in the abstract.

It is a narrower claim:

- PostgreSQL is the better default for this repository right now
- the project should favor the backend that is easiest to operate, model, inspect, and tune during fast iteration
- if scale pressure later proves that default wrong, the benchmark history remains available for reconsideration

## Architecture Boundaries

The shared crawler boundary from `BNDD-0005` still stands:

```text
React / Desktop UI / Future agents
  -> app-facing client contract
  -> HTTP API or thin app adapter
  -> shared crawler workflow in crates/btc-network
  -> repository trait
  -> PostgreSQL adapter in crates/btc-network-postgres
  -> PostgreSQL
```

Rules:

- do not move protocol parsing or session logic into storage adapters
- do not put SQL in React, CLI entrypoints, or Tauri command layers
- keep shared read and write traits stable unless a real query gap appears

## Default Backend Direction

The canonical implementation defaults are:

- new default adapter crate: `crates/btc-network-postgres`
- SQL layer: `sqlx` with explicit SQL
- pool: `sqlx::PgPool`
- migration binary: `apps/postgres-migrate`
- default crawler and API wiring: PostgreSQL
- default local infra and Make targets: PostgreSQL

Explicit non-goals for this slice:

- no `TimescaleDB`
- no ORM
- no generic `storage_backend` runtime option
- no dual-write transition
- no first-class equal support for PostgreSQL and ClickHouse

## Storage Model

The default PostgreSQL schema shape is:

- append-first `node_observations`
- append-only `crawler_run_checkpoints`
- forward-only `schema_migrations`

Latest-state analytics use plain relational SQL:

- `DISTINCT ON` for latest-row-per-key selection
- composite indexes for endpoint-latest, run-latest, and recent run-scoped reads
- no separate mutable latest-state table in this first slice
- no extension-specific hypertables, compression, or columnstore features

## Replacement and Preservation Plan

This replacement plan is implemented in the repository:

1. author `BNDD-0007` and mark `BNDD-0002` and `BNDD-0005` as historical rather than default
2. add the PostgreSQL adapter crate, migrations, and integration tests
3. add the PostgreSQL migration binary and switch `apps/crawler` and `apps/api` to PostgreSQL
4. switch default local Docker and Make workflows to PostgreSQL

Cutover rules:

- no historical data migration is required
- the project is still experimental, so a fresh-start backend replacement is acceptable
- future reconsideration of ClickHouse requires a new BNDD or an explicit update to this one

## Verification

- `cargo test -p btc-network`
- `cargo test -p btc-network-postgres`
- `cargo test -p btc-network-api`
- `cargo test -p btc-network-crawler`
- `make test`
- `make security`

Required scenarios:

- PostgreSQL migrations apply idempotently
- observation streaming preserves stable observation and batch identifiers
- latest checkpoint and startup-recovery semantics stay aligned with the shared workflow
- analytics reads keep the current browser-safe API contract
- crawler and API config parsing use PostgreSQL defaults

## Revisit Conditions

Revisit this decision when one of these becomes true:

- PostgreSQL operational cost or query behavior becomes the main bottleneck
- the current latest-state queries become too awkward or too slow at real scale
- storage growth clearly exceeds the intended single-VM operating model
- the product requires a broader analytical workload that justifies reactivating the ClickHouse path
